# 原型 Skill 流程改造设计方案（多工具版）

## 0. 文档定位

这是一份**待评审的设计方案**，不涉及任何代码/skill 文件改动。内容汇总了本轮讨论的全部结论：

- `01`/`02` 号文档的问题分析
- `03` 号文档核实源码后发现的差距（A/B 路线等机制从未真正写进 skill、组件库未回流、范围收敛未真正生效）
- `04` 号文档的分层方案（输入分诊 / 批量分诊 / 预期管理 / 执行调度）
- 本轮讨论新增的三条修正（控制权在人、给样板与拆分帮助、会话质量管理）
- 本次新增的多工具适配设计（Claude → Kimi → Codex → DeepSeek）

评审通过后，再落地成对 `prototype-guide`、`prototype-site`、`工具无关-编排提示词.md` 的具体文件改动。

## 1. 总体架构回顾（本次改造不动的部分）

- **`prototype-site`**：生成引擎。产出承载 `data-page/data-action/data-nav/data-modal` 等 IR 标记的高保真静态原型，是研发转 Vue 的知识来源，职责不变。
- **`prototype-guide`**：产品经理入口。把业务语言翻译成生成规格，隐藏技术细节，本次改造的绝大部分内容都加在这一层。
- **`工具无关-编排提示词.md`**：跨工具搬运层。已确立"单一真源、只编排指路、不复制内容"的原则（详见第 9 节）。

本次改造在这三者现有的职责边界内做加法，**不新增独立 skill**——避免"该用哪个 skill"变成产品经理的新门槛。

## 2. 治理元规则（贯穿全部改造，优先级最高）

这是本轮讨论确认下来、必须写进 `SKILL.md`/`flow.md` 最前面的一条元规则，所有后面的机制都要服从它：

> **skill 负责把信息对齐、把风险讲清楚、给出可执行的建议、主动提供样板和拆分帮助；要不要收窄范围、要不要一次冲到底、要不要清会话、要不要走某条有代价的路，永远由使用者决定。skill 不能以"我判断你这样不好"为理由单方面拒绝执行，也不能在对方已经表态后反复劝说。**

具体拆成三条可执行的子规则：

1. **只讲一次**：任何风险提示（目标不现实、输入过载、批次将尽）只完整讲一次，讲清楚代价，不重复唠叨。
2. **表态即执行**：使用者选定后立刻照办，包括"我知道有风险，别废话，全做"这种选择；照办的同时在交付说明里留痕"按您的要求，已提示风险后执行"。
3. **脏活自己扛**：批量拆分、交接文档、样板生成，这些是 skill 主动做的工作，使用者可以选择跳过（"别拆了，你看着办"），但默认应该替他把这些工序先做出来，而不是让他自己去悟"正确用法"。

后面第 4~7 节的三个机制（输入分诊、两道门、拆分与批次），全部在这条元规则下设计——即所有的"检查"都只产出**信息**，不产出**拒绝**。

## 3. 输入分诊层（新增，位于 `flow.md` 步骤 0 之前）

在开场立约之前，先判断这次输入属于哪一类，决定走哪条后续流程：

| 输入形态 | 判断依据 | 走的流程 |
|---|---|---|
| 口述 / 单份文档 / 单张截图，目标做 1 个新页 | 默认场景，材料单一 | 不变，走现有 `flow.md` |
| 1~2 个旧原型页面/截图，仅作参照 | 参照物数量少，且用户说的是"做这一页" | 现有"已有页面还原"能力，不用大改 |
| 整份旧原型目录 / 多份文档混合 / "你自己看着办" | 出现目录级输入，或一次给了 ≥3 类不同性质的资料（PRD、编码表、旧原型、其他） | **触发第 4 节"两道门"+ 第 5 节"拆分机制"** |

判断本身不需要精确算法，用"文件数量、材料类型数"这类可以直接观察到的信号即可，具体阈值见第 4.2 节。

## 4. 两道门：进入生成前必须过、但只提示不拦截

这两道门的共同点：**都在读取任何文件全文之前先做侦察，都只产出诊断报告，不自行决定拒绝执行。**

### 4.1 门一：目标兼容性检查

检查用户表达的目标，和 `prototype-site` 的产出能力之间是否存在结构性矛盾。做成一份可持续扩充的**能力边界清单**，不止"旧原型"一种场景：

| 用户目标里出现的信号 | 与 skill 能力的矛盾点 | 需要摆出的代价 |
|---|---|---|
| "和旧原型一模一样""像素级还原""完全复刻" | `prototype-site` 是统一规范生成引擎，不是复刻引擎 | 给 A（默认风格，代价：不像）/ B（提炼风格，代价：只能接近，耗时更长）/ C（直接复用旧文件，代价：丢失 IR 标记，研发不能直接用）三条路 |
| "换一套完全不同的 UI 风格/组件库" | 组件库是预置的，替换是工程排期，不是运行时能做到的事 | 说明"临时换风格 = 只能做到接近，正式换风格需要另立项集成" |
| （后续遇到新的矛盾场景，持续往这张表里加） | | |

命中任意一条，必须在**开始生成之前**把代价讲清楚、给出选项，用户选哪条都执行——这是门一，不是关卡。

### 4.2 门二：资源过载检查

**核心原则：先侦察，不读全量。** 在读取任何文件的完整内容之前，先用便宜的操作（`ls`/`find`/`dir`/文件大小，视工具的 shell 能力而定，见第 9 节的降级路径）拿到量化信号，而不是把所有文件读进上下文之后才发现太大。

| 信号 | 低风险 | 中风险 | 高风险 |
|---|---|---|---|
| 页面级源文件数量 | ≤5 | 6~15 | >15 |
| 材料类型混杂数（PRD/编码表/旧原型/其他同时出现几类） | 1 类 | 2 类 | ≥3 类 |
| 总文本体量（估算） | 小 | 中 | 明显偏大（本次实测的 34 个 html + 两万字 PRD 属于这一档） |

命中高风险，先输出诊断报告（第 4.3 节），停在这一步等用户表态，**不是默认拒绝**——用户一句"别管了，全部做"，立刻执行，同时留痕。

这一层的价值不是消灭崩溃（工具本身稳定性不在 skill 控制范围内），而是消灭"心知肚明还硬冲、事后才复盘发现早该提醒"这种最没意义的失败模式。

### 4.3 诊断报告：两道门共用的输出格式

不管命中哪道门，统一用这个结构，信息要素不能少：

> **我看到的输入**：（材料清单 + 数量/体量的具体数字，两道门都要报数字，不说"感觉有点多"这种模糊话）
> **这样输入会带来的问题**：（会不会超出单次任务安全范围 / 会不会和你要的结果本身矛盾，讲清楚具体是哪一类）
> **我的建议**：（给可执行的最小修正动作——见第 5 节的"样板与拆分"，不是一句"请规范输入"）
> **需要你确认的点**：（明确等一个选择，选完立刻执行）

## 5. 拆分与样板机制（对齐 `xdocs/批量优化页面/prompts/` 的既有实践）

这一节不是凭空设计，是把你在石油焦掺配模型批量优化项目里已经验证过的做法，原样搬进 `prototype-guide`：

- 该项目每一页有独立的 `prompts/BMGYxx-页面名.md`，30 行左右说清楚一页的查询字段、表格列、操作、交互——这就是"正确输入的样板"该长的样子，不用另造格式。
- 批次划分是 1~4 页一批（第 0 批 1 页，第 1/2/3 批各 4 页），和 `04` 号文档建议的"3-5 页/批"基本吻合，说明这个批次大小是实测出来的经验值，不是拍脑袋。

**拆分流程**（命中门二的高风险、且用户同意走这条路时触发）：

1. 提议："要不要我先把这份大文档/整个目录，按页面拆成一份份独立、干净的小提示词（就像你们石油焦项目 `prompts/` 目录那样），你确认清单没问题，我们再一页页做？"
2. 用户同意后，skill 消化原始材料（PRD 定业务口径、编码表定范围、旧原型定视觉参照），产出 N 份单页提示词文件到约定目录（如 `<站点目录>/prompts/<编码>-<页面名>.md`），每份包含：页面编码、页面名称、页面类型、菜单归属、查询字段、表格列、顶部/行操作、交互要求。
3. 用户过一遍这些小文件（比读原始大文档轻松得多，这一步本身就是"确认范围"），勾出本批要做的 3-5 页。
4. 之后喂给 `prototype-site` 的永远是这份干净的单页文件，不是原始的 PRD/编码表/旧原型全文——**这一步顺带解决了"不要把脏乱材料直接怼给生成引擎"这个问题，消化工作在 `prototype-guide` 这一层做完，不传导下去。**

用户当然可以跳过这道工序（"别拆了，你自己看着办全部做"），跳过就直接按原始材料做，风险自负，在交付说明里留痕即可——符合第 2 节的元规则。

## 6. 批次与会话管理机制（解决"批量质量下降 vs 手动清会话更好"）

这是本轮讨论新增、也是最关键的一块。核心思路：**把"每一页任务输入是否新鲜"和"要不要物理重开会话"拆成两件独立的事，前者是无条件的硬规则，后者是免费提供、但控制权在用户手上的选项。**

### 6.1 硬规则（不管是否清会话都要做）：每页生成前重新读取本页任务包

每一页生成时，永远从第 5 节产出的独立提示词文件（或交接文档里记录的该页任务描述）重新读取，不依赖对之前几十轮对话的"记忆拼凑"。石油焦项目的 `prompts/*.md` 本身就是这么用的——每页任务的输入质量不随会话变长而衰减，因为衰减的根源是"靠记忆回忆细节"，不是"会话长"这个事实本身。这条即使用户坚持不清会话、一路做到第 30 页，也能兜住大部分质量。

### 6.2 每批（3-5 页）结束，固定产出/更新一份交接文档

格式参照 `xdocs/批量优化页面/Claude接手说明.md` + `执行台账.md` 的形态，包含：

- 已完成页面清单 + 产物路径
- 本批的重要修正/踩坑记录（这一条同时是 `03` 号文档"组件库回流"的天然入口——写交接文档时顺手把这批用到的、之前组件库没覆盖的样式记下来）
- 待办页面清单
- 下一步建议
- 当前工作树的改动概览（哪些文件被动过）

有了这份文档，"要不要清会话"从"一件需要手动重新组织材料的麻烦事"，变成"点一下新会话，让它先读这份文档"——重开的成本趋近于零。

### 6.3 是否重开会话：控制权在用户，skill 只负责建议

每批结束，主动问一句"要不要开个新会话继续，我这边进度都记好了"，用户说继续就继续，不反复劝。

### 6.4 折中方案：不想重开会话的人，每批边界重读一遍规则文件

参照 `原型skill使用与改进建议.txt` 里"Kimi 适配项"已经提出的思路：每做完一批，强制重新读一遍 `SKILL.md`/`references/flow.md` 的关键规则，作为比"物理重开会话"更轻量的刷新手段，对抗长会话的规则遗忘。

## 7. 组件库回流与产物自检（衔接 `03` 号文档，不重复设计）

`03` 号文档已经排好优先级：

- **P1**：把这次实测修复的样式类（`.title-badge`、`.st-count`、`.page-header` 等）回流进 `skills/prototype-site/assets/framework/assets/app.css`——目前仍是 0 处命中，需要执行。
- **P2**：给 `theme.css` 加机械校验（grep 检查是否混入 `display/grid/position` 等布局规则），写进 `prototype-site` 自检清单。

本方案不重复设计，但第 6.2 节的交接文档，天然是让 P1 持续发生的载体——每批交接文档里记的"本批修正"，定期人工誊抄进 `app.css` 即可，不用另外发起一次专项梳理。

## 8. 决策总览（第 3~7 节的关系）

```
输入进来
  │
  ▼
【第3节】输入分诊 ── 单页/少量参照 ──▶ 走现有 flow.md，不受本次改造影响
  │
  ▼ 批量/混合输入
【第4节】两道门（目标兼容性 + 资源过载）──▶ 输出诊断报告，等用户表态
  │
  ▼ 用户表态：接受拆分建议
【第5节】拆分成单页提示词文件 + 用户勾选批次范围
  │
  ▼
【第6节】按批（3-5页）执行：每页读独立任务包 → 每批完更新交接文档 → 问要不要清会话
  │
  ▼
【第7节】组件库/自检（持续动作，不阻塞流程）
```

## 9. 多工具适配设计

### 9.1 现状与既有原则（不重新发明）

`prototype-guide/测试与交付/03-多工具准备手册.md` 已经把跨工具的核心原理定清楚了，本次改造沿用，不重新设计：

- **单一真源**：`SKILL.md` + `references/*.md` 这套 markdown 不因为工具不同而改内容。
- **Claude Code**：原生识别 `SKILL.md`，自动加载，是基线工具。
- **其他工具没有"skill"概念**，只有"项目级指令/规则/自定义提示"，本质是让它们"挂载同一套 markdown"，而不是另写一套规则。
- **万能保底**：不管工具机制如何，"会话开头把工作交底文本贴进去"这条路一定能走通。

本次新增的机制（输入分诊、两道门、拆分、批次交接）要按同一原则融入，**只更新"工作交底文本"和 `工具无关-编排提示词.md`，不新增另一套规则文件**。

### 9.2 目标工具矩阵与已知差异

| 工具 | Skill/项目指令承载方式 | 文件/Shell 工具能力 | 已知特点或风险 | 本次新增机制的适配要点 |
|---|---|---|---|---|
| **Claude Code**（基线） | 原生 `SKILL.md` 自动发现加载 | 完整（Read/Glob/Grep/Bash 等） | 规则遵从度高，是本次改造的验证基线 | 两道门的"侦察"直接用 Bash/Glob 等工具跑出精确数字，无需降级 |
| **Kimi CLI** | 项目指令文件机制待逐工具确认，保底是会话开头粘贴工作交底文本 | 已验证具备本地文件读取能力（实测里读取了整个旧原型目录），Shell 能力待确认 | 实测中出现过异常退出、死机；`原型skill使用与改进建议.txt` 指出"中文长规则黏着度更好"，需要精简 | 交接文档必须是**独立文件**而非依赖对话记忆（Kimi 会话不稳定，文件是唯一可靠的状态载体）；工作交底文本里两道门的规则要写成简短祈使句中文，不要长段抽象描述 |
| **Codex** | `AGENTS.md` 自动加载，或 `~/.codex/prompts/` 自定义命令 | 具备本地文件读写和 Shell 能力 | 机制较明确，风险相对低 | 两道门的侦察可直接用 shell 命令；`AGENTS.md` 里追加本次新增机制的编排段落 |
| **DeepSeek**（新增，尚未实测） | 未知，按"最保守假设"设计：先按"会话开头粘贴工作交底文本"验证，机制确认后再补充专属挂载方案 | **未知，需要第一步验证**：是否能读本地文件、是否能执行 shell 命令 | 无实测记录，风险未知 | 见 9.4 节的降级路径——这是本次新增工具里唯一需要"先验证能力、再决定用哪条路"的一个 |

> DeepSeek 相关信息目前没有实测依据，矩阵里的判断是"按最保守假设先设计降级路径，真机验证后再回填"，不作为已确认事实对待。

### 9.3 分层承载策略

沿用现有三层结构，本次新增内容也分三层落：

1. **单一真源层**：`skills/prototype-guide/references/input-intake.md`（新增"输入分诊"“两道门”“拆分流程”三节）、`references/flow.md`（新增"批次与会话管理"一节）、`skills/prototype-site/SKILL.md`（自检清单新增交接文档/组件库校验条目）。这一层只写一次，所有工具共享。
2. **跨工具编排提示词层**：`skills/prototype-guide/工具无关-编排提示词.md` 的【启动提示词】里，在现有"步骤 5 生成硬门禁"之前，插入"步骤 4.5：两道门检查"和"步骤 2.5：输入分诊"的编排指令（只讲"做什么、去读哪个参考文件"，不复制 `input-intake.md` 的内容原文，保持单一真源）。
3. **工具专属注意事项层**：`测试与交付/03-多工具准备手册.md` 里给每个工具补一条"本次新增机制的注意事项"，即上表最后一列的内容，不散落到别处。

新增一个工具（比如以后要上 Trae 或别的），只需要在第 3 层加一节，不动第 1、2 层——这是维持"单一真源"的关键约束。

### 9.4 降级路径：工具没有可靠的文件/Shell 能力时怎么办

第 4.2 节的"资源过载检查"依赖能跑 `ls`/`find`/`dir` 拿到精确数字。如果目标工具（典型是 DeepSeek，或 Kimi 在某些托管环境下）经确认**不具备**这类工具调用能力，只能退化成"直接问用户要数字"，而不是让模型凭上下文里贴的内容自己数——因为凭感觉数容易数错，尤其是长文档粘贴进对话之后：

> "为了不让处理过程中途卡死，能不能先告诉我：这次一共涉及几个页面文件？大概多少字/多少张截图？"

这条降级路径要写进"多工具准备手册"，作为每个新工具接入前必须先确认的第一件事（和现有 3.1 节"Kimi 机制确认"的做法一致）：**先探测工具的文件/Shell 能力，再决定两道门的侦察方式走"自动数"还是"问用户"。**

### 9.5 规则表达上的跨模型鲁棒性

结合 `原型skill使用与改进建议.txt` 的经验（"Kimi 对中文长规则黏着度更好"）和本次两道门设计的特点，给规则写法定一条通用原则：

- **能用数字判断的，就不要用感觉判断**——"文件数 >15" 比 "输入感觉很多" 更容易被弱模型稳定执行，这也是第 4.2 节阈值表设计成具体数字、而不是模糊描述的原因。
- **规则用简短祈使句中文**，避免大段解释性文字——解释放在这份设计方案和 `references/*.md` 里给人看，工作交底文本和门二的判断规则要写成"跑 xxx → 结果 >15 → 停下报告"这种可直接执行的短句。
- **交接文档是跨工具最可靠的状态载体**——不管工具的对话记忆机制多不同，只要它能读写文件，交接文档就能生效，这是本次设计里唯一不依赖"工具聪明程度"的机制，优先级最高。

## 11. 验证方式

在 `test-01/手动测试手册.md` 现有 R/A/B/D 用例基础上，补充两类，且都要至少在 **Claude + 1 个非原生 skill 工具**（建议先选 Codex，机制最明确）上各跑一遍：

- **两道门用例**：分别构造"目标不兼容"（要求像素级一致）和"资源过载"（>15 个文件）两种输入，验证是否输出统一格式的诊断报告、用户表态后是否立刻执行、是否有反复劝说的情况（这是违规项）。
- **批次交接用例**：构造一个能拆出 6-8 页的批量输入，验证第 3-5 页后是否主动提出交接文档 + 是否询问要不要开新会话；新开一个会话只喂交接文档，验证能否正确接续而不重新问一遍需求。
- **工具降级用例**（仅 DeepSeek 或其他未验证工具）：先确认其文件/Shell 能力，验证降级路径（问用户要数字）是否被正确触发，而不是模型自己瞎猜数字。

## 12. 分期实施建议

改动面不小，建议也按"批次"的哲学来推进，不要求一次性重写全部文件：

1. **第一期**：L4（组件库回流）+ L7（交接文档模板）——不依赖其他改动，收益直接，先做。
2. **第二期**：L1 + L2——把输入分诊、两道门、拆分流程、批次会话管理写进 `prototype-guide` 本体，在 Claude Code 上完整验证一轮（对应第 11 节的两道门用例、批次交接用例）。
3. **第三期**：L5——把第二期验证过的机制编排进跨工具提示词，先在 Codex 上验证（机制最明确，风险最低）。
4. **第四期**：L6——补 Kimi 的适配细节，并对 DeepSeek 做第一次能力确认（读文件？跑 shell？），确认结果回填矩阵表后再决定挂载方式。
5. **第五期**：L3——`prototype-site` 自检清单收尾，全流程跑一遍完整回归。

每一期结束都建议单独评审一次产物，而不是等五期全做完再看——这也是第 6 节"批次+交接"这套方法论对我们自己实施过程的一次自我应用。
