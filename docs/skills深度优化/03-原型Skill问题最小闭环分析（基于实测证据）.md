# 原型 Skill 问题最小闭环分析（基于审计监督模型实测证据）

## 一、这份文档是干什么的

前两份文档（`01-原型Skill使用反馈与优化方向总结.md`、`02-全量原型输入实践复盘与批量迁移风险分析.md`）已经把机制性问题分析清楚了；`test-01/原型skill使用与改进建议.txt` 是你上周模拟产品经理实测后的复盘，结论也已经很完整。

这份文档不重复前面的内容，只做两件事：

1. 把你提的 4 个问题，和已有的两份分析、这次实测的证据对应起来，避免"问题、原因、证据"分散在三份文档里对不上。
2. 我去核对了 skill 源文件和实测产物之后，发现有 3 个前面材料都没有明确点出、但对下一步优先级判断很关键的事实，需要单独说清楚。

结论先说：**这次实测暴露的问题，根因不是模型能力不够，而是"哪些行为该由 skill 硬编码保证、哪些行为这次只是临场发挥"没有分清楚。** 好的行为（盘点、A/B 选择、分批、留痕）这次都出现了，但去检查 skill 源文件后确认——**它们目前一条都没有写进 skill 里**，出现只是因为这次会话被人工照着测试脚本临场引导。换一个不熟悉脚本的产品经理，或者换一个工具，大概率不会自动复现。

## 三、核对源文件后新增的 3 个关键发现

这 3 点是我对照 `skills/prototype-guide/`、`skills/prototype-site/` 的实际源文件，以及 `test-01/result/` 的实测产物逐条核实后确认的，之前三份材料都没有把这层"文档说的和代码里有的对不对得上"讲清楚。

### 发现 1：A/B 双路线、盘点、偏离提醒——目前一条都没有写进 skill 里

`test-01/手动测试手册.md` 的标题是"改造后手动测试手册"，`原型skill使用与改进建议.txt` 里也写"已实现'盘点-风格差异分析-A/B双路线'和'偏离提醒-确认-留痕'"。但实际检查：

- `skills/prototype-guide/SKILL.md`、`references/input-intake.md`、`references/flow.md` 里**没有**任何"旧原型目录输入""A/B 路线""批量确认"相关规则。`input-intake.md` 里唯一相关的是"已有系统截图→按已有页面还原"，只覆盖单张截图场景，不覆盖"整个旧原型目录"。
- 全局搜索"路线 A / 路线 B / 双路线 / 基准页"等关键词，命中的**只有** `02-全量原型输入实践复盘与批量迁移风险分析.md` 这份分析文档本身，skill 源文件里一处都没有。

也就是说：这次实测里出现的盘点、A/B 二选一、样板页确认、偏离留痕，全部是**你和 Kimi CLI 在会话里临场演出的**（很可能是照着手动测试手册的脚本一步步引导出来的），而不是 skill 固化的行为。这解释了为什么 02 文档观察到的"B 路线边界不够硬"——因为这条边界本来就没有写进任何规则文件，模型没有可依据的硬约束，全凭上下文里你的引导。

**这是当前最容易被忽视、但优先级最高的一条**：01/02 两份文档提的"建议方向"里最核心的一条（场景识别 + 批量模式），到今天还是"建议"，没有变成 `SKILL.md` 或 `references/*.md` 里的一条规则。

### 发现 2：这次修复的样式缺陷，没有回流进 skill 共享的 app.css

`test-01/原型skill使用与改进建议.txt` 第二部分第 1 条明确写了"把这次修复的所有样式回流进 skill"，`result/prototype-progress.md` 也详细记录了 5 类修复（`.title-badge`、`.st-count`、`.btn-warning/.btn-danger`、`.page-header`、`.dash-grid-21`、`.field-block` 等）。

实际检查 `skills/prototype-site/assets/framework/assets/app.css`（共 147 行）：以上关键类名**一个都不存在**。也就是说这次实测定位到的"组件库覆盖不足"问题，修复只停留在 `test-01/result/` 这个一次性产物里，还没有变成 skill 的永久能力。下次产品经理再用 skill 做类似页面（驾驶舱统计卡、状态徽章、textarea 表单），会**重新踩一遍同样的坑**。

### 发现 3：范围失控风险这次没有触发，是因为目录恰好是空的，不代表问题已解决

`应用模块划分与编码.md` 里写得很清楚："现阶段聚焦 4 个一级模块"（风险管理/任务中心/模型中心/系统管理），`首页/AI应用中心/知识库/数据集成中心` 明确标注为"后续阶段扩展预留"。实际检查旧原型目录：

- 8 个一级模块文件夹里，`AI应用中心`、`数据集成中心`、`系统管理`、`知识库` 这 4 个是**完全空的文件夹（0 个文件）**。
- 有内容的只有 `风险管理`(7页)、`模型中心`(15页)、`任务中心`(11页)、`首页`(1页)，合计 34 个页面文件，和 `prototype-progress.md` 里"已生成 7 页 + 待生成 27 页 = 34 页"的计划完全对上。

这说明 02 文档担心的"读取整个目录会把预留模块也纳入范围"这次**没有实际发生**——但不是因为 skill 识别出了"这 4 个模块是预留的、不该做"，而是因为这几个文件夹本来就是空的，模型无内容可读。这是运气，不是防线。如果下次预留模块的文件夹里已经有几页占位原型（很常见的情况），同样的风险大概率会兑现。

更值得注意的是**范围收敛这一层本身也没有生效**：编码表虽然把范围锁定到"4 个一级模块"，但没有进一步限定"这个批次具体做哪几页"。模型实际的行为是把这 4 个模块下**能看到的全部 34 个页面**都纳入了计划（`prototype-progress.md` 的待办清单就是这 34 页减去已完成的 7 页），而不是像 02 文档建议的那样先产出页面清单、让产品经理勾选 3-5 页。也就是说，即便这次的输入材料已经算是"角色分工相对清楚"的一份（编码表限定范围、PRD 说明业务口径、原型给视觉参考），**模型依然是把目录里现存的文件数量当成本次范围，而不是把"本批次要做的页面"当成范围**。这印证了 02 文档十一节提出的建议——"页面计划清单 + 产品经理勾选"这一步必须是硬规则，光靠输入材料本身写得规范是不够的。

## 四、最小待办（按这次证据的支撑强度排序，不是按理论重要性）

- **P0：把"批量输入检测 → 强制盘点 → 页面清单勾选（默认 3-5 页/批）"写进 `skills/prototype-guide/SKILL.md` 和 `references/input-intake.md`。**
  这是三份材料唯一都指向、且被两轮独立验证（01/02 的理论分析 + 这次实测）反复证实、但至今没有任何代码/文档改动落地的一条。不需要新起一个 skill，在现有 `input-intake.md` 里加一节"批量/旧原型目录输入的识别与处理"即可，把手动测试手册里 A1/B1/D3 用例已经验证过的对话流程，原样固化成规则文字。

- **P1：把 `test-01/result/prototype-progress.md` 里记录的组件修复，回流进 `skills/prototype-site/assets/framework/assets/app.css`。**
  这一步你在 txt 总结里已经想到了，只是还没做。工作量小（对照 5 类样式补丁）、收益直接（下次任何页面都不会再踩同样的坑），建议优先于新流程设计先做掉。

- **P3：把 `test-01/原型skill使用与改进建议.txt` 第一部分（PM 正确用法六条）整理成一页纸，随 skill 使用手册分发。**
