# 原型 Skill 流程整体优化方案：批量诉求与 AI 工具约束的调和

## 一、这份文档要回答什么

你已经把架构和这次失败的直接原因看得很准了：

- `prototype-site` 是真正的生成引擎，产出的原型里带着 `data-page/data-action/data-nav/data-modal` 这些研发要用的 IR 标记，是"承载研发知识"的那一层；
- `prototype-guide` 是给不懂技术的产品经理用的引导层，把业务语言翻译成 `prototype-site` 能吃的规格，本意是降低门槛；
- 这次产品经理把旧 HTML + 需求文档 + 一堆"他觉得有用的资料"一次性丢进去、还要求产出和旧原型"一模一样"——这确实是输入类型不对、噪声污染、单次工作量过载、目标本身不现实，四个问题叠在一起的必然失败，你已经预判到了。

你要我做的是往前一步：**从流程上重新设计，让 skill 能够引导产品经理走到正确路径上，同时把"AI 工具本身的约束"（单任务要小、要干净、连续执行会话会累积污染）也内化成 skill 自己的硬规则**，而不是每次都靠事后教育产品经理。你也提到这是一对矛盾——产品经理天然想要"能不能批量"，而 AI 工具天然要求"任务要小、要独立"。

下面先审你的判断，再把这对矛盾拆开来看，最后给出分层的流程设计和落地清单。

## 二、先审一遍你的判断

大部分是对的，但有三处需要补充或修正，会直接影响方案设计的方向：

**1.（认可）输入类型错配 + 噪声污染 + 目标不现实，是这次失败的直接原因。**
这个判断准确，不需要修正。`03-原型Skill问题最小闭环分析.md` 里已经用实测证据核对过：`prototype-guide` 目前确实只设计了"文本 + 截图"两种输入通道，从来没有一条规则覆盖"整份旧原型目录"这种输入形态，失败是设计空白导致的必然结果，不是模型偶发失误。

**2.（需要修正）"退出或死机"不完全是提示词设计问题，不能只靠"控制输入"来解决。**
你把"崩溃/死机"和"信息污染、噪声过大"放在同一条里，但这其实是两类不同性质的风险：

- 一类是**质量风险**——输入噪声大、范围不清，会让生成结果跑偏、后面页面质量下降。这类可以靠"控制输入、锁定范围"来根治。
- 另一类是**稳定性风险**——长会话本身的上下文体积、工具（Kimi CLI）自身的资源占用和 bug，即使输入干净、范围锁得再死，只要任务链条足够长，依然有非零概率崩溃或卡死。这类**只能靠"断点续做"来兜底，不能靠"不出错"来预防**。

这个区分很重要：如果方案只做"控制输入"，你会把"进度文件 + 断点续做"这类兜底机制设计成可有可无的附加项；但实际上不管输入多干净，只要是长任务，这个兜底机制都必须**默认存在**，因为你没法把崩溃概率降到零。

**3.（你已经点到、但需要单独立项）"同一会话连续做多个独立任务，会不经意上下文污染"，这一点现有 skill 完全没有对策。**
`prototype-guide` 现有的"逐页推进"规则（`flow.md` 步骤 7：做完一页问要不要做下一页）解决的是"范围不要一次太大"，但**没有解决"会话本身在变长"**——哪怕严格一页一页做，做到第 20 页时，上下文里已经堆了前面 19 页的全部对话和产物细节，02 文档观察到的"后生成页面质量下降"，本质上就是这个问题的体现，而不是范围控制失败的体现。这两个问题需要两条不同的规则来治，不能指望"一页一页做"顺带解决"会话变长"。

## 三、把"矛盾"拆成两条独立的轴，而不是一个问题

你说这是矛盾的综合体——产品经理想要简单粗暴的批量，AI 工具需要小而干净的任务。这个矛盾是真实的，但它可以拆成两条相互独立的轴，分别有对策，拆开后就不矛盾了：

| 轴 | 风险 | 对策方向 | 谁来扛 |
|---|---|---|---|
| **范围/噪声轴** | 一次给的材料太多、太杂，模型把无关内容也纳入生成范围 | 强制盘点 + 资料角色分工 + 范围锁定 | skill 的**入口规则**（输入分诊） |
| **会话/上下文寿命轴** | 同一会话越做越长，即使范围控制得再好，后面的页面依然会漂移；且工具本身有非零崩溃概率 | 批次执行 + 进度文件 + 主动切分会话 | skill 的**执行调度规则** |

这两条轴治好之后，"产品经理要的批量"和"AI 工具要的小任务"就不再互斥了——**产品经理要的其实不是"一次性把所有页面塞给 AI"，而是"我不想每页都盯着、我希望这件事自己往前推进"**。这是体验诉求，不是执行方式的诉求。真正的解法是把这两者解耦：

> **对产品经理暴露"批量"的体验，对 AI 工具内部坚持"小任务+新鲜会话"的执行方式。复杂度由 skill 自己吸收，不转嫁给产品经理，也不违反 AI 工具的工作方式。**

具体怎么做，见下面第四节。

## 四、分层方案设计

### 第 0 层：输入分诊（新增，插在 `prototype-guide` 最前面，比 `flow.md` 步骤 0 还早一步）

现在 `flow.md` 一上来就是"步骤 0：开场立约"，默认输入就是"文本/截图找单页"。需要在这之前加一个分诊动作，先判断这次输入属于哪一类，再决定往下走哪条流程：

| 输入形态 | 判断依据 | 走的流程 |
|---|---|---|
| 纯口述 / 单份文档 / 单张截图，目标是做 1 个新页 | 现有默认场景 | 不变，走现有 `flow.md` |
| 1~2 个旧原型页面或截图，仅作为"这页长什么样"的参照 | 参照物数量少，且产品经理明确说的是"做这一页" | 现有"已有页面/截图还原"能力（对应 `input-intake.md` 的"金矿"分类），不用大改 |
| 整份旧原型目录 / 多份文档混合 / 产品经理说"你把这些都读一下自己判断" | 出现目录级输入，或一次给了 3 种以上不同性质的资料 | **触发第 1 层"批量分诊流程"**，禁止直接进入 `flow.md` 步骤 3（选模板/生成） |

这一层的作用是把"这次到底是新建单页，还是旧原型批量迁移"这个判断，从"模型隐式猜测"变成"skill 显式分类"——03 文档发现的问题之一，就是现在完全没有这一步，模型只能凭上下文自己猜。

### 第 1 层：批量分诊流程（新增，对治"范围/噪声轴"）

一旦第 0 层判定为批量输入，强制先做以下几步，**这几步本身不生成任何页面**：

1. **资料盘点**：列出收到的每一份材料是什么（PRD/编码表/旧原型目录/其他），用大白话回吐给产品经理，附一句"这是我读到的，对吗"。
2. **资料角色判定**：固定优先级——编码表/范围文档决定"这批做哪些页"，PRD 决定业务口径，旧原型决定视觉和结构参照，目录里其他没被点名的内容一律当背景、不生成。这一条 02 文档已经提出，之前一直停留在建议，没有写进 `input-intake.md`，这次需要真正落地成文字规则。
3. **页面清单 + 范围锁定**：输出候选页面清单，标注"本期/预留/不确定"，默认给产品经理**勾选 3-5 页**作为第一批，而不是默认全量。这一步是本次实测证明"即使编码表已限定范围，模型依然会把目录里现存的全部文件当成计划范围"这个问题的直接解药——把范围锁定从"依赖模型自觉"变成"必须有一次显式勾选动作"。
4. **旧原型场景下的预期管理**：见第 2 层。

如果产品经理坚持"不勾选，全部一次做完"，允许执行，但要先把话说清楚（见下面话术），并在交付物里留痕"这是偏离推荐流程的批量草稿"——这部分 `手动测试手册.md` 的 D3 用例已经设计过对话脚本，只是从来没有变成 skill 规则文字，这次要把它搬进 `input-intake.md` 里固化。

### 第 2 层：预期管理——"和旧原型一模一样"这个目标本身要在生成前被拦下来

这次失败里最容易被忽视的一点：产品经理要的"一模一样"，不是输入问题，是**目标设定问题**，必须在生成之前、而不是生成之后才告诉他做不到。

当第 0 层识别到"旧原型输入"，在进入任何生成步骤之前，必须先说清楚三条路，而不是只讲 01/02 文档里已有的 A/B 两条：

- **A（默认风格）**：快、稳、和平台统一，但长相和旧原型不一样。
- **B（提炼旧风格）**：尽量接近旧原型的视觉，但**只能做到"接近"，明确不能保证像素级一致**——这句话必须由 skill 主动说在前面，不能等产品经理验收时才发现"怎么不一样"。
- **C（如果产品经理仍然坚持"必须一模一样"）**：明确告诉他这已经超出"AI 重新生成"这条路能做到的范围，只有"直接复用旧文件、跳过 prototype-site 的规范"能做到形似，但代价是这份原型不再具备转 Vue 所需的 IR 标记和统一结构，研发没法直接用——这个代价必须讲清楚，让产品经理自己决定要不要豁免生成质量换取视觉一致。**不能既要"和旧原型一模一样"又要"研发能直接用"，这两者在旧原型不是按 prototype-site 规范做的前提下，本身互斥。**

这一步的本质，是把"目标不现实"这件事从"生成完了才发现失败"提前到"生成之前就摆明代价"，避免又一次把不可能的目标交给模型硬做、然后模型只能悄悄打折扣交差。

### 第 3 层：执行调度——批次执行 + 进度文件 + 主动切分会话（对治"会话/上下文寿命轴"）

这一层解决的是你点出的"同一会话连续做多个独立任务会不经意上下文污染"，现有 skill 完全没有对策，需要新增：

1. **进度文件是标准产物，不是运气好才用上的东西**。这次实测里 `prototype-progress.md` 救了两次崩溃，但它是临场生成的，不是 skill 规定必须产出的。需要写进 `prototype-site` 的自检清单：只要是超过 1 页的任务，第一步就要建立/更新进度文件，记录已完成页、待办页、资料角色判定结果、已选路线（A/B/C）。
2. **每做完一批（3-5 页），主动提议切分会话**，而不是无限期在同一个会话里往后堆。话术类似：
   > "这一批 3 页都做完了，进度我已经记在 `prototype-progress.md` 里。接下来的几页建议你开一个新的对话——这样我读取进度文件重新对齐后开始，不会因为这个会话已经聊了很久而把前面页面的细节带偏。你也可以选择就在这继续，我会留意状态。"
3. **新会话开场，第一件事是读进度文件对齐**，而不是重新问一遍需求。这一条要写进 `flow.md` 的"步骤 0"，作为"如果检测到已有进度文件"时的分支开场白。
4. 这一层不承诺"不会崩溃"，只承诺"崩了/卡了，新开会话读进度文件就能接着做，不用从头再来"——这是兜底，不是预防，要在文档和对产品经理的话术里都讲清楚,避免产品经理误以为"用了这套流程就不会再崩"。

### 第 4 层：产物自检与组件库沉淀（衔接 03 文档，不重复展开）

`03-原型Skill问题最小闭环分析.md` 里已经排好了这部分的优先级（组件库回流进 `app.css`、`theme.css` 布局规则机械校验），这份文档不重复，只强调一点衔接关系：**第 1~3 层解决的是"流程有没有跑对"，第 4 层解决的是"跑对了之后细节像不像"**——两者都要做，但这次的"流程整体优化"重点是前三层，第 4 层按 03 文档的 P1/P2 顺序推进即可。

## 五、"能不能批量"的最终答案

把上面四层合起来，给产品经理的答案应该是：

> **"可以批量，你说一次做完，我按 3-5 页一批帮你往下推，每批做完我给你看一眼、留个进度记录，你确认没问题我接着做下一批，中途工具偶尔抽风崩了也不用担心，我读进度文件接着来。你不需要每一页都守在这，但每一批我都会跟你对一下。"**

对产品经理来说，这就是"批量"——他不需要每页都参与决策，整个过程感觉是自动往前推进的。对 AI 工具来说，实际执行的仍然是"一页一页生成、3-5 页一批、批与批之间可能切会话"——完全符合"单任务要小、要独立、会话不要无限变长"的约束。**矛盾消失的关键，不是在"批量"和"逐页"之间选一个,而是让产品经理感知到的颗粒度（批次）和 AI 实际执行的颗粒度（单页）不是同一个颗粒度,中间那层翻译由 skill 来做。**

## 七、怎么验证方案有效

在 `test-01/手动测试手册.md` 的基础上补两类用例，专门验证这次新增的部分（原有 R/A/B/D 用例继续保留）：

- **批次切分用例**：给一份能拆出 6-8 页的批量输入，验证是否在第 3-5 页后主动提出"建议开新会话"，以及新会话开场是否会先读进度文件。
- **目标不现实用例**：明确要求"必须和旧原型像素级一致"，验证是否会说清楚 A/B/C 三条路和 C 的代价，而不是默默按 B 做然后交出一个"看起来差不多"的结果。

这两类用例目前的测试手册里都没有覆盖，是这次实测暴露出来、但还没有被验证过的部分。
