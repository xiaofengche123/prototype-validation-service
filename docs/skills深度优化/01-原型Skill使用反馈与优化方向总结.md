# 原型 Skill 使用反馈与优化方向总结

## 一、背景

近期产品经理在使用 `prototype-guide` 和 `prototype-site` 两个原型相关 skill 时，反馈生成效果不理想。经过初步复盘，我们判断问题不完全是生成能力本身不足，而是当前 skill 的设计场景与部分实际使用场景不匹配。

目前需要把问题、原因和后续解法同步给产品经理、项目经理和相关负责人，避免继续用不合适的方式使用 skill，也为后续优化明确方向。

## 二、当前两个 Skill 的原始定位

### 1. prototype-guide：面向“从零生成新页面”

`prototype-guide` 的设计场景是：

- 产品经理正在和客户沟通需求；
- 当前还没有明确成型的参考页面；
- 产品经理用业务语言描述要做什么页面；
- skill 帮助产品经理梳理需求、检查信息是否充分、建议页面类型；
- 确认后再交给 `prototype-site` 生成单个静态原型页。

也就是说，`prototype-guide` 更适合解决“我现在只有需求，帮我画一个新页面”的问题。

### 2. prototype-site：内置一套默认原型页面风格

`prototype-site` 的设计场景是：

- 根据整理好的页面需求生成 PSM / 绿星链通风格的静态原型页面；
- 页面结构、菜单注册、交互标记、表格渲染、弹窗、按钮、布局约束等都按它自己的规则执行；
- 页面视觉风格主要由它内置的样式和模板决定。

因此，`prototype-site` 不是一个“任意页面复刻工具”，而是一个“按统一规范生成新原型页面”的工具。

## 三、这次反馈不佳的核心原因

产品经理实际使用时，存在一种当前 skill 没有充分覆盖的场景：

> 产品经理原来已经做过一套原型，现在希望把已有页面转换成 `prototype-site` 产出的统一风格页面。

这个场景和原始设计场景不同。

原始设计场景是“从需求生成新页面”；现在遇到的是“把已有原型迁移成统一规范页面”。两者的输入不同，目标也不同。

如果产品经理把已有页面、截图或旧原型直接交给当前 skill，当前 skill 很容易把它当成普通需求材料处理，而不是当成“需要迁移和保留结构的源页面”。这样生成结果就可能出现偏差。

## 四、具体偏差表现

当前 skill 在处理已有原型时，可能出现以下问题：

- 只抽取字段、按钮、表格等业务元素，但没有保留原页面的信息结构。
- 过度套用 `prototype-site` 默认布局，导致原页面的重点区域、分组方式、操作路径发生变化。
- 没有明确判断“这是新建页面需求”还是“已有页面转化需求”。
- 没有让产品经理提前确认：是保留原页面布局，还是接受统一风格重构。
- 对已有页面中的视觉重点、业务优先级、模块层级识别不足。
- 生成结果看起来像“重新画了一版”，而不是“把原页面转成统一规范的一版”。

因此，产品经理认为效果不好是合理的。问题根源不是简单的“提示词写得不够细”，而是当前 skill 缺少“已有原型迁移”这个工作模式。

## 五、结论

本次分析结论如下：

1. 当前 `prototype-guide` 和 `prototype-site` 可以支持“从文字需求生成新页面”。
2. 当前两个 skill 没有完整支持“已有原型页面迁移 / 复刻 / 风格归一化”。
3. 如果把已有原型当成普通需求输入，生成结果大概率会偏离产品经理预期。
4. 后续需要把“新建页面”和“已有页面转化”拆成两个不同流程处理。

一句话概括：

> 现在的问题不是产品经理不会用，而是产品经理实际遇到的“旧原型转新规范”场景，超出了当前原型 skill 的原始设计边界。

## 六、建议的优化方向

### 1. 在入口处先判断使用场景

后续 skill 应先询问或自动判断当前任务属于哪一类：

- 新建页面：只有业务需求，需要从零生成页面。
- 旧页面转化：已有截图、HTML、Axure、墨刀、蓝湖或其他原型，需要迁移成统一规范页面。

不同场景走不同流程，不能混用。

### 2. 为“已有原型转化”增加专门流程

对于已有原型转化，应增加以下步骤：

- 先识别旧页面的模块结构、主次关系、字段、按钮、表格、弹窗和跳转。
- 再确认哪些内容必须保留，哪些可以按统一风格调整。
- 然后按 `prototype-site` 的规范重建页面。
- 最后输出“保留项”和“调整项”清单，方便产品经理确认。

### 3. 明确保留策略

在转化旧页面时，需要让产品经理确认保留程度，例如：

- 尽量保持原页面布局，只做风格统一。
- 保留业务结构，但允许重新整理布局。
- 只保留业务字段和操作，页面按新规范重新设计。

这个选择必须由产品经理确认，不能由模型默认决定。

### 4. 增加一个独立的转化类 Skill 或模式

建议新增或扩展一个面向“已有原型转化”的能力，例如：

- `prototype-convert`
- `prototype-migration`
- 或在 `prototype-guide` 中增加“旧原型转化模式”

该能力不再只问“你要做什么页面”，而是先问“你希望原页面哪些部分被保留”。

## 七、后续使用建议

在优化完成前，建议按以下方式使用：

- 如果是全新页面需求，继续使用 `prototype-guide`。
- 如果已有旧原型，不要直接要求“照这个生成一页”。
- 应先说明：“这是旧原型转化，不是新建页面。”
- 同时补充希望保留的内容，例如页面结构、字段、按钮、表格、筛选区、弹窗、跳转关系等。
- 对视觉和布局要求高的页面，建议先让 skill 输出“页面理解摘要”，确认后再生成。

## 八、下一步计划

建议后续对两个 skill 做一次深度优化：

1. 更新 `prototype-guide`，增加场景识别：新建页面 / 旧原型转化。
2. 补充旧原型转化的需求确认流程。
3. 更新 `prototype-site`，增加“按源页面结构重建”的生成约束。
4. 增加转化结果自检清单，重点检查是否保留了原页面结构和业务重点。
5. 准备一到两个旧原型样例，作为验证用例，确保优化后效果可被产品经理稳定复用。

这次优化的目标不是让页面“看起来更炫”，而是让 skill 更准确地区分任务类型，减少误用，让产品经理能够稳定得到符合预期的原型结果。
