# 全量原型输入实践复盘与批量迁移风险分析

## 一、背景

在前一次问题复盘中，我们已经确认：当前 `prototype-guide` 和 `prototype-site` 更适合“根据需求从零生成新页面”，对“已有原型页面迁移为统一规范页面”的支持不足。

本次进一步复盘产品经理的真实使用方式后，发现还有一个更关键的问题：产品经理并不是一页一页给参考页面，而是把 PRD、编码表和整个旧原型目录一次性交给模型，让模型读取目录下所有原型页面后重新生成。

这类输入方式会显著增加大模型理解和生成的不确定性，也是本次生成质量不稳定、范围膨胀、执行时间过长的重要原因。

## 二、产品经理的实际操作方式

产品经理描述的操作大致如下：

1. 把需求文档、编码表、旧原型页面目录一起交给 Kimi CLI。
2. 没有按页面逐个指定参考原型。
3. 直接要求模型读取目录下所有原型页面。
4. 期望模型能够参考旧原型样式，重新生成新的 prototype-site 站点页面。

沟通中确认：

- 提问：你一次性把所有页面都给了它吗？
- 产品经理回答：是的，我就让它读目录下的所有原型页面。

这说明产品经理的真实诉求不是“生成一个新页面”，而是“把一整套旧原型批量迁移为新的统一站点”。

## 三、初步分析

这类使用方式本身存在较大风险。

大模型在一次性读取大量页面、PRD、编码表和其他资料时，容易出现以下问题：

- 不同页面之间的信息相互干扰。
- 参考页面、需求页面、未来规划页面混在一起。
- 模型无法稳定判断哪些页面是本次必须生成，哪些只是背景材料。
- 页面风格、布局结构、业务字段、菜单范围容易相互污染。
- 后续生成页面的质量会随着上下文变长而下降。
- 执行时间显著增加，异常退出概率上升。

因此，产品经理反馈效果不好，并不是单纯因为模型“不会画页面”，而是输入方式超出了当前 skill 的稳定处理边界。

## 四、昨日改造思路

为了验证产品经理的真实用法，曾尝试对原型 skill 做过一次改造，目标是让 skill 支持两种输入：

### 1. 原有输入方式

产品经理只有文字需求或需求文档，没有现成原型页面。

此时继续走原有流程：

- 收集业务需求；
- 检查信息是否充分；
- 建议页面类型；
- 逐页确认；
- 按 prototype-site 默认风格生成页面。

### 2. 新增输入方式

产品经理同时提供：

- 旧原型页面；
- PRD；
- 编码表；
- 其他辅助资料。

此时不应直接按默认模板生成，而应先识别这是“已有原型迁移”场景，再让产品经理选择处理路线。

## 五、A/B 双路线测试设计

为了验证这次改造，手动测试手册中设计了 A/B 两条路线。

测试手册位置：

`test-01/手动测试手册.md`

### 路线 A：平台默认风格

适用场景：

- 产品经理提供了旧原型，但只是作为业务结构参考；
- 最终页面仍希望按 prototype-site 内置的统一默认风格生成。

期望行为：

- 先盘点旧原型页面数量和用途；
- 分析旧原型和默认风格差异；
- 主动让产品经理在 A/B 两条路线中选择；
- 产品经理选择 A 后，继续按正常逐页流程生成；
- 业务口径以 PRD 为准，页面长相只参考旧原型，不照搬旧页面。

### 路线 B：按旧原型提炼统一风格

适用场景：

- 产品经理希望保留旧原型那套视觉风格；
- 同时又希望页面符合 prototype-site 的站点结构和交互规范。

期望行为：

- 让产品经理指定 1 到 2 页作为视觉标准；
- 先生成站点级 `assets/theme.css`；
- 先做 1 个样板页给产品经理确认；
- 产品经理认可后，再继续做后面的页面；
- 如果视觉不对，优先调整 `theme.css`，而不是在每个页面里写独立样式。

## 六、本次实验输入

本次实际验证的是路线 B。

输入材料包括：

- PRD：`test-01/审计监督模型_2026年6月30日/审计监督模型/绿星智审-审计监督模型_PRD_需求规格说明书_V1.md`
- 编码表：`test-01/审计监督模型_2026年6月30日/审计监督模型/应用模块划分与编码.md`
- 旧原型目录：`test-01/审计监督模型_2026年6月30日/审计监督模型/`

执行方式：

- 使用 Kimi CLI；
- 按产品经理真实方式，让模型读取整个旧原型目录；
- 选择路线 B，即不使用 prototype-site 默认页面风格，而是尽量按照产品经理提供的旧原型样式重新生成页面。

执行情况：

- 总执行时间约 2 小时；
- Kimi CLI 异常退出 2 次；
- 过程中电脑出现 1 次死机；
- 生成菜单和计划页面数量明显超过预期。

## 七、实际产物观察

输出目录：

`test-01/result/`

观察到的关键现象：

1. 旧原型目录中包含大量 HTML 页面，约 34 个页面级文件。
2. `prototype-progress.md` 中记录：已生成 7 页，待生成 27 页。
3. 实际 `menu.config.js` 中只注册了已生成的 7 页。
4. 这说明模型内部已经计划按全量旧原型规模推进，但实际生成只完成了一部分。
5. 产物中已经生成了 `assets/theme.css`，并记录了基准页来源。
6. 但 `theme.css` 中出现了较多布局相关规则，例如 `display`、`grid`、`width`、`position` 等。

这些现象说明：模型不仅在提炼视觉风格，也在试图补一套通用组件和布局能力，已经超出了“主题皮肤文件”的原始边界。

## 八、主要问题判断

### 1. 输入范围失控

产品经理一次性提供了整个旧原型目录，模型会倾向于把目录下所有页面都视为本次要处理的范围。

即使编码表中写了“现阶段聚焦 4 个一级模块”，材料中仍然包含首页、AI 应用中心、知识库、数据集成中心等后续预留或附加内容。模型如果没有强约束，很容易把这些内容也纳入计划。

### 2. 任务颗粒度过大

一次性迁移几十个页面，对大模型来说不是一个稳定的单步任务。

它需要同时完成：

- 读取旧页面；
- 理解 PRD；
- 对照编码表；
- 判断生成范围；
- 提炼视觉风格；
- 生成站点框架；
- 写主题文件；
- 逐页重建页面；
- 注册菜单；
- 保持交互和跳转一致。

这些任务全部压在同一次会话里，失败概率很高。

### 3. 资料角色不清

在这类混合输入中，不同资料应该有不同优先级：

- 编码表：决定本次页面范围和页面编码；
- PRD：决定业务口径、字段、流程和功能；
- 旧原型：决定页面结构、视觉参考和交互形态；
- 后续规划或附加目录：只能作为背景，不应默认生成。

如果 skill 不先明确这些资料角色，模型就会把它们混在一起使用。

### 4. 上下文污染明显

多个旧页面同时进入上下文后，页面之间的字段、按钮、布局、菜单、状态和跳转会互相影响。

最终结果可能表现为：

- 某个页面出现其他页面的功能；
- 菜单比原型或本期范围多很多；
- 页面命名和编码不一致；
- 页面结构越来越偏离基准页；
- 后生成页面质量下降。

### 5. B 路线边界不够硬

路线 B 的原始目标是“提炼旧原型统一风格”，但实际执行中，`theme.css` 里出现了大量布局规则。

这会带来后续维护风险：

- `theme.css` 不再只是主题皮肤，而变成第二套布局框架；
- 页面级结构问题可能被主题文件掩盖；
- 后续某页调整可能影响全站；
- 与 prototype-site 原有 `app.css` 的职责边界变模糊。

## 九、研究结论

本次实践证明：

1. 产品经理真实使用场景不是单页新建，而是整套旧原型迁移。
2. 当前 skill 如果直接接收“PRD + 编码表 + 全量旧原型目录”，很容易发生范围膨胀。
3. 一次性读取所有旧页面，会导致上下文污染，降低生成质量。
4. 路线 B 是有价值的，但必须增加更严格的流程控制。
5. “旧原型转化”不能简单作为原有流程的一个补丁，而应设计成独立的迁移流程。

一句话概括：

> 这次实验不是证明路线 B 不可行，而是证明路线 B 不能直接从“全量目录输入”开始生成，必须先锁定范围、确认基准、做样板页，再逐页推进。

## 十、建议的流程改造

### 1. 增加强制盘点阶段

当输入中包含旧原型目录时，skill 不应立即生成页面。

第一步只能做盘点：

- 识别收到哪些资料；
- 统计旧原型页面数量；
- 按目录列出页面清单；
- 标记哪些看起来是本期页面，哪些可能是后续预留；
- 让产品经理确认“这些页是否都需要处理”。

### 2. 增加强制范围锁定

在生成前必须让产品经理确认范围。

建议默认规则：

- 默认一次只做 1 页；
- 一次最多允许选择 3 页作为小批量；
- 如果产品经理要求全量生成，必须进入“批量草稿模式”，并明确提示质量和返工风险。

### 3. 明确资料优先级

skill 应固定采用以下优先级：

1. 编码表决定页面范围和编码；
2. PRD 决定业务口径；
3. 旧原型决定结构和视觉参考；
4. 目录中的其他页面默认只作为背景，不自动生成。

### 4. 路线 B 拆成两阶段

路线 B 不应直接批量生成所有页面。

建议拆成：

第一阶段：主题样板阶段。

- 选择 1 到 2 个基准页；
- 生成 `theme.css`；
- 生成 1 个样板页；
- 让产品经理确认风格。

第二阶段：逐页迁移阶段。

- 每次只迁移 1 页；
- 每页都明确源原型页、目标编码、页面名称、参考 PRD 章节；
- 每页完成后让产品经理确认，再进入下一页。

### 5. 严格限制 theme.css 职责

`theme.css` 应只负责：

- 颜色；
- 字体；
- 圆角；
- 阴影；
- 表头、按钮、标签、卡片等皮肤类覆盖。

不应包含：

- 页面布局；
- `display/grid/flex` 主结构；
- `position` 定位；
- 大范围 `width` 控制；
- 只服务某一个页面的选择器。

如果确实需要补充布局能力，应放回页面结构或 prototype-site 的基础样式设计中，而不是混进主题文件。

### 6. 把“全量生成”定义为偏离路线

如果产品经理坚持“全部一次做完”，skill 可以执行，但必须先提示：

- 页面数量多，生成质量可能下降；
- 菜单和页面范围可能需要二次校验；
- 后续返工风险由该选择带来；
- 本次结果定位为“批量草稿”，不是正式高保真交付。

确认后再执行，并在交付说明中记录这是偏离推荐流程的结果。

## 十一、后续测试建议

为了验证优化是否有效，建议补充以下测试：

1. 同一批输入材料，只选择 1 个页面生成，观察质量是否稳定。
2. 同一批输入材料，一次选择 3 个页面生成，观察是否出现交叉污染。
3. 同一批输入材料，要求全量生成，明确标记为批量草稿，记录质量下降点。
4. 对路线 B 的 `theme.css` 做自动检查，统计是否出现布局类规则。
5. 对菜单数量做检查，确认生成菜单不超过产品经理确认的范围。
6. 对每页生成前后做“源页面 → 目标页面”映射表，确保没有凭 PRD 扩张出额外页面。

## 十二、最终建议

建议后续不要把“旧原型迁移”作为 `prototype-guide` 原流程里的一个简单分支，而应作为一个更严格的能力来设计。

推荐命名可以是：

- 旧原型迁移模式；
- 原型复刻模式；
- 原型风格归一化模式；
- 或独立 skill：`prototype-migration`。

该模式的核心不是“更快生成更多页面”，而是：

- 先控制输入；
- 再锁定范围；
- 再统一风格；
- 最后逐页迁移。

只有这样，才能让产品经理的真实使用方式变得可控，也才能让生成结果稳定达到可交付水平。
