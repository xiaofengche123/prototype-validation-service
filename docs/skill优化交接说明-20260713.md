## 二、优化背景（一段话版）

同事用旧版 skill 批量处理"旧 HTML 原型 → 新站点页面"时，全站 14,681 行旧原型只还原出 1,872 行（平均还原率 13%），18 页完全缺失、5 页严重缩水，事后逐页人工修复耗费大量时间。根因：①旧 HTML 被压缩成一句话/30 行摘要再生成（信息压缩比 30:1）；②skill 引用的《05-HTML页面输入处理规范.md》根本不存在；③内嵌组件被误做成独立页面；④生成后无量化核对。**结论（使用者拍板）：质量第一——第一版质量差，后面调优就脱离 skill 管线，总成本更高。**

## 三、三轮优化都改了什么

### 第 1 轮：补上缺失的翻译层规范

- **新建 `skills/prototype-site/references/html-input-spec.md`**（旧 HTML 输入的唯一执行规范，最重要的文件）：
  - §3 数量守恒提取：从源页面**数出**查询字段数/表格列数/数据行数/页签数/弹窗数/交互点数，写进提示词头部
  - §3.3 业务说明段：页面性质五选一（配置工具页/录入页/查询页/工作台/看板），防止"评分卡编辑→被做成打分页"这类语义跑偏
  - §4 内嵌组件识别：含 `window.self !== window.top` iframe 检测脚本或"内嵌-"文件名前缀 → 是嵌入组件，**不生成独立页面、不注册菜单**，整合进宿主页（弹窗须补触发按钮、状态 Tab 须补对应 mock 数据）
  - §5 编码兼容：新编码 `^[A-Z]{2}[A-Z]{2}[0-9]{2}$`（6 位 2+2+2）；旧系统沿用编码允许 `^[A-Z]{2,6}[0-9]{2}$`（如 `FXGLQD01`），**不强行改 6 位**
  - §7 生成后量化自检：列数/字段数/页签数/弹窗数不一致 → 强制重做；新页面 < 源页面 50% 行数 → 缩水警戒，禁止直接交付
- **prototype-site/SKILL.md**：app.css 允许在末尾 `/* === 站点扩展区（只增不改） === */` 追加新组件类（抽屉/开关/雷达图等），绝不修改既有规则，新增类记入 handoff.md
- **prototype-guide/references/input-intake.md**：删除"每份提示词约 30 行"上限（这是压缩根因）；iframe 信号优先于"有完整外壳"的独立页判断
- **check-site.js 修复**：C4 正则字面量里 `${target}` 不插值的 bug；C1/C3/C4 兼容旧编码；C9 增加弹窗反向检查（`data-modal` 容器必须有触发入口）

### 第 2 轮：转 Vue 链路联动

- **模块/子模块推导规则变更**（prototype-to-vue3 + prototype-vue3-framework 及其 references）：
  - 旧规则"取编码前 4 位"作废（会把 `FXGLQD01` 切成无意义的 `FX/GL`）
  - **新规则：取编码字母部分（去掉末尾 2 位数字），前 2 字母=模块，其余字母=子模块**。`BMSP01→BM/SP`，`FXGLQD01→FX/GLQD`
  - 转换时**不得把旧编码改名成 6 位**（会切断 data-nav 目标和旧系统对应关系）
- **样式扩展移植**（新硬规则）：转 Vue 前必须对照站点 app.css"站点扩展区"和 theme.css，页面用到的扩展类必须在 Vue 侧有定义——单页专用写页面 scoped 样式，多页共用追加到框架 `prototype-parity.scss` 末尾扩展区（只增不改）。否则构建通过但视觉静默降级
- **整站转换范围收窄**：只转 `menu.config.js` 已注册的页面；`pages/` 下未注册的"内嵌-"参考遗留文件**跳过**，不转成独立 Vue 页

### 第 3 轮：质量优先模式（本轮最重要，针对旧 HTML 输入）

使用者确认：质量 > 速度 > token 消耗。工具按 Kimi/Codex/Claude 三者通用设计。

1. **双输入生成**：翻译层落盘两份文件——
   - `prompts/<编码>-<页面名>.md`：详细提示词（结构索引 + 核对清单）
   - `prompts/<编码>-<页面名>.source.html`：剥壳后的源页面业务内容（**细节兜底：选项值/mock 数据/交互逻辑/视觉效果直接从这里抄**，禁止凭"常见做法"补位）
   - 生成每页时**两份都必须读入**。禁止为省 token 跳过源文件
2. **大页面分块对照**：剥壳后 ≤800 行整页读入；>800 行按区块切分（查询区/统计卡/标签区/表格区/弹窗区/脚本区）逐块对照生成再拼装，**脚本区必须单独整块读入**。800 行是按三工具最小上下文取的保守值；确认上下文充足（如 Claude）可放宽到 1500 行，但出现任何核对 ✗ 立即回退 800
3. **提示词质量门（生成前必过）**：新脚本 `skills/prototype-site/assets/check-prompts.js`，校验 6 项——编码格式、剥壳源文件配套存在、数量守恒每项有具体数字、长度比例下限（源 >300 行→提示词 ≥100 行；>800 行→≥200 行）、业务说明段五选一、模糊词黑名单（"若干/多列/下拉筛选不带选项/同旧版/常见做法"出现即不合格）。**不达标退回重新提取，禁止带病进入生成**
4. **批次收紧（HTML 输入模式）**：每批 2~3 页（覆盖通用的 3~5 页）；剥壳后 >800 行的复杂页单页一批；**首批只做 1 页**，交付对比报告全 ✓ 且使用者认可还原度后才扩批
5. **交付对比报告（每页必附）**：输出"源页面 vs 生成页"逐项对照表（查询字段数/表格列数/数据行数/页签数/弹窗数/按钮数/图表数/交互点，✓/✗），写进交付说明并追加到 progress.md。**带 ✗ 不许交付**

## 四、你（Kimi/Codex/Claude）执行时的操作规程

### 收到旧 HTML 原型输入时（核心场景）

```
1. 先读 skills/prototype-site/references/html-input-spec.md（唯一执行规范）
2. 内嵌组件识别（iframe 检测脚本/"内嵌-"前缀）→ 命中则整合进宿主页，不做独立页
3. 完整读取源 HTML 全文（禁止只读开头猜内容）
4. 剥壳 → 落盘 prompts/<编码>-<页面名>.source.html
5. 数量守恒提取 → 写 prompts/<编码>-<页面名>.md（含业务说明段）
6. 过质量门：
   - 有 Node 环境（Codex/Claude CLI）：站点根目录运行
     node <本仓库>/skills/prototype-site/assets/check-prompts.js prompts/<编码>-<页面名>.md [...]
     修完所有 ✗ 才继续
   - 无运行时（Kimi 纯对话）：人工逐项过 html-input-spec.md §3.5 检查表，
     并在 progress.md 写"质量门已过（人工）"
7. 双输入生成：同时读提示词 + source.html；>800 行分块对照
8. 生成后：对照数量守恒清单逐项核对（§7），跑 check-site.js（有 Node 时）
9. 交付：附对比报告（全 ✓），更新 progress.md / handoff.md
```

### 批次与状态纪律（所有工具一致）

- 目标目录已有 `prompts/*.md` / `handoff.md` / `progress.md` → **读取后接续，不得重新拆分、不得重做已完成页**。文件是唯一可信状态源，对话记忆不可靠
- HTML 输入批次：首批 1 页 → 全 ✓ → 2~3 页/批；复杂页（>800 行）单页一批
- 每批结束更新 handoff.md（已完成/待办/踩坑）+ progress.md（执行日志/偏差清单）

### 两个自检脚本（有 Node 环境时都要跑）

| 脚本 | 时机 | 作用 |
|---|---|---|
| `skills/prototype-site/assets/check-prompts.js` | 提示词写完、生成开始前 | 质量门：拦截摘要式/含模糊词的提示词 |
| `skills/prototype-site/assets/check-site.js` | 每页生成后 | 站点校验：编码/菜单/IR 标记/弹窗配对/外壳残留 |

两个脚本都在**站点根目录**运行，exit 0 才算通过。

### 转 Vue 时（prototype-to-vue3）

- 只转 menu.config.js 已注册页面；模块目录按"字母部分切分"推导
- 检查页面用到的 app.css 扩展区/theme.css 类，逐个移植到 Vue 侧（scoped 或 parity 层扩展区），否则视觉降级
- 旧编码保持原样，不改 6 位

## 五、红线清单（违反任何一条 = 本页重做）

1. ❌ 把旧 HTML 压成一句话/短摘要再生成（30 行以内的提示词 = 不合格输入）
2. ❌ 生成时只读提示词、不读 source.html（双输入是强制的）
3. ❌ 凭"常见做法"补位源页面里明明有的细节（选项值/列名/数据必须从源里抄）
4. ❌ 把含 iframe 检测脚本或"内嵌-"前缀的文件做成独立页面/注册菜单
5. ❌ 带 ✗ 的对比报告直接交付，让使用者自己发现问题
6. ❌ 为省 token / 赶速度跳过质量门、跳过分块对照、一批塞太多页
7. ❌ 修改 app.css / prototype-parity.scss 的既有规则（扩展区只增不改）
8. ❌ 把旧编码（7~8 位）强行改成 6 位
9. ❌ 目录里已有 prompts/ 和 handoff.md 时重新拆分或重做已完成页面
