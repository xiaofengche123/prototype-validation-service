# 模板提炼 skill（template-distill）— 使用手册

> 给使用者看的操作手册。重点是**大量可拷贝的提示词案例**——复制下来，改【】里的内容就能用。

---

## 1. 这个 skill 帮你做什么

给它一个「风格来源」（三选一：跑得起来的旧 Vue 系统 / UI 设计交付物 / 高质量静态原型页），它产出一个**风格包草案**：

```text
kimi-skill-project/style-packs/<pack-id>/
  contract.md            ← 视觉契约：所有风格数值的唯一出处
  prototype/
    app.css              ← 原型侧样式（契约的投影）
    page-template.html   ← 原型业务页模板
    golden/xxx.html      ← 你亲自确认过的黄金样板页
  parity-checklist.md    ← 数值核对清单（后续防漂移的硬门）
  CHANGELOG.md
  sources/               ← 取值证据（截图、量取 JSON、确认卡）
```

这个包随后交给「模板校准」skill（template-calibrate），实例化出新风格的三件套（prototype-site / prototype-vue3-framework / prototype-to-vue3 新家族）并分批调优到收敛。

**它不做的事**：不生成 Vue 侧样式（校准第0批做）、不做业务页面、不改现有三件套。

---

## 2. 开工前准备（三条路径各自的清单）

| 路径 | 你要准备好的东西 |
| --- | --- |
| A 旧Vue系统 | 系统能跑起来（本地或测试环境地址）；能打开 DevTools；最好能用浏览器自动化工具 |
| B UI设计稿 | 设计图文件（png/figma导出）；文字描述；低质量 HTML（如有，仅当语义参考） |
| C 静态原型 | 给定的原型页面目录；确认哪些页面属于这套风格 |

先想好 `pack-id`（英文 kebab-case，如 `audit-dark`、`cnal-green`），想不好让 skill 建议。

## 3. 提示词案例库（复制即用）

### 案例 A1：路径A 完整开工（从旧 Vue 系统提取）

```
用「模板提炼」skill 从旧系统提取风格包。

pack-id：【audit-dark】
来源类型：路径A（旧Vue系统）
源系统地址：【http://localhost:8080】（技术栈：【Vue2 + Element UI】）
可用登录：【admin / xxxx】

要求：
1. 打开源系统的这些代表页做取值和截图（按7类页面原型归档到 sources/）：
   - 标准列表页：【用户管理】
   - 多区块输入页：【立项申请】
   - 图表分析页：【风险驾驶舱】
   - 审批流转页：【审批中心】
   - 配置矩阵页：【权限矩阵】（没有的类型标"不适用"）
2. token 数值必须从真实页面 computed style 量取（用 skill 自带的 measure-tokens.js），
   不许只看源码 CSS；每个值在契约里标 measured:computed-style。
3. Element UI 上有而 AntD4 复刻不了或不值得复刻的组件，逐条写进契约 §6 兼容性裁决表，
   裁决「复刻/近似/放弃」，近似的写清差异，最后列给我逐条确认。
4. 用契约做出 prototype/app.css 和 page-template.html，
   再做一张黄金页（业务场景用【用户管理】），完成后给我出黄金页确认卡，等我确认。
```

### 案例 A2：浏览器不可用时的降级取值

```
继续路径A提取，但浏览器自动化不可用。降级方案：
1. 我手动打开源系统页面，你告诉我 measure-tokens.js 怎么用，我把控制台输出的 JSON 粘给你；
2. 我截图发你，你从截图量取相对值，量不准的（阴影、精确px）标 filled:补位 待我确认；
3. 所有降级取的值在契约出处里如实标注，不许标成 measured:computed-style。
```

### 案例 A3：兼容性裁决表专项

```
针对风格包【audit-dark】做兼容性裁决专项：
把源系统里这些组件逐个和 AntD4 对照：【树形穿梭框、可编辑表格、步骤条带副标题、
带汇总行的表格、日历热力图】。
每个给出：AntD4 对应组件 / 复刻可行性 / 建议裁决 / 若近似则差异是什么。
输出成契约 §6 的表格行，我来逐条确认后你再写进契约。
```

### 案例 B1：路径B 完整开工（UI 设计稿）

```
用「模板提炼」skill 从设计交付物提取风格包。

pack-id：【cnal-green】
来源类型：路径B（UI设计稿）
设计资产：
- 设计图：【xdocs/资料/新版设计/首页.png、列表页.png、表单弹窗.png】
- 文字说明：【xdocs/资料/新版设计/设计规范说明.md】
- 低质量HTML：【xdocs/资料/新版设计/demo.html】

硬要求（skill 规则，这里重申）：
1. demo.html 只准当语义参考（页面上有什么元素），它的 CSS 一个值都不许进契约；
2. 顺序必须是"先黄金页、后契约"：先照设计图做一张黄金页给我确认，
   确认通过后才反向提炼契约；
3. 设计图没画的（hover、空态、加载态、表格密度、断点、超长截断）逐条列成
   §7 缺口清单，每条给出你的补位建议和依据，等我逐条确认；
4. 设计图只覆盖了【列表页和表单弹窗】两类，其余5类页面原型的样式外推规律
   写进 §8，外推逻辑要能讲清楚。
黄金页业务场景用【设计图里的列表页对应的业务】。
```

### 案例 B2：黄金页返工（路径B 常见循环）

```
黄金页第【2】轮返工：
1. 【查询区控件比设计图高，设计图量出来是36px，你做成了32px】——改契约 §3 表单控件行；
2. 【状态标签设计图是描边样式不是填充】——改 §4 并重做标签；
3. 【卡片阴影太重】——设计图量不出精确值，给我3档候选（轻/中/重）截图对比，我选。
改完更新黄金页，重新出确认卡。
```

### 案例 B3：缺口与外推专项确认

```
把风格包【cnal-green】契约 §7/§8 整理成确认清单发我：
§7 每条缺口按「缺口 / 你的补位决策 / 依据」列表；
§8 每条外推规则附一个具体例子（比如"图表分析页的卡片头会长什么样"）。
我确认哪条你落哪条，没确认的不许当成已定。
```

### 案例 C1：路径C 完整开工（高质量静态原型）

```
用「模板提炼」skill 从给定静态原型提取风格包。

pack-id：【audit-blue】
来源类型：路径C（高质量静态原型）
给定原型目录：【kimi-skill-project/test-01/审计监督模型_2026年6月30日/】

要求：
1. 给定页面只当视觉验收基准，不直接当模板；
2. 挑信息密度最高的一张（建议【任务管理列表】）做结构归一化：
   剥外壳、表格改数据驱动、打 data-action/data-modal/data-nav、
   换 prototype-* 结构类、去内联样式；
3. 归一化后和原页并排目视比对，视觉不许有损失，差异列给我看；
4. 从给定 CSS 提炼契约：合并重复值、归纳 token、扔掉一次性 hack，
   数值以页面实际渲染为准，出处标 given:prototype-css；
5. 归一化后的那张就是黄金页，出确认卡等我确认。
```

### 案例 C2：路径C 归一化质量抽查

```
对归一化产物做抽查：
1. grep 黄金页确认：无 style= 内联、无手写重复 <tr>、无外壳类（top-header/sidebar/page-tabs）；
2. 列出所有 data-action/data-modal/data-nav 及其对应行为桩；
3. 原页和归一化页并排截图（或我人工开两个窗口对比），把你发现的所有视觉差异列成表，
   标注每条是"故意的（进契约§6）"还是"要修的"。
```

### 案例 D：混合来源裁定

```
我们同时有旧系统和新设计图。用「模板提炼」skill 开工前先做来源裁定：
- 设计图是【新风格的权威】，旧系统只提供【页面类型样本和业务语义】；
- 契约 §0 风格真相裁定写清楚：视觉数值一律来自设计图（路径B规则），
  页面原型分类和代表页截图来自旧系统（路径A的盘点动作）；
- 两边冲突时以设计图为准，冲突项记录在 sources/ 里备查。
按这个裁定走路径B主流程。
```

## 4. 三条路径的关键纪律（为什么手册反复强调）

| 纪律 | 违反后果（真实教训） |
| --- | --- |
| A：只认 computed style，不认源码 | 源码样式有覆盖链，读出来的不是用户看到的；契约错一个值，校准阶段全线返工 |
| B：低质 HTML 的 CSS 禁入契约 | 低质 HTML 是别的工具随手生成的，颜色间距全不可信，抄进来等于污染唯一数据源 |
| B：先黄金页后契约 | 设计图欠规格，直接提契约会把空白留给以后每次生成随机填，页面之间就不一致了 |
| C：给定页面不当模板 | 视觉高质量≠结构高质量：内联样式、手写 tr 会让转 Vue 无法机械化 |
| 全路径：出处标注 | 校准阶段要靠出处决定评审权重；没有出处的值出了问题无法追溯 |
| 全路径：确认门 | 现行家族质量提升的一半来自执行者当裁判；跳过确认门=后面几十页建立在没验收的地基上 |

## 5. 常见问题

**Q：pack-id 怎么起？**
英文 kebab-case，「产品线/客户-主色或主题」，如 `lxlt-blue`、`audit-dark`。定了就别改，它会出现在版本戳、目录名、校准工作目录里。

**Q：黄金页选什么业务场景最好？**
信息密度最高的标准列表页：有统计卡、查询区、状态页签、批量栏、表格（含状态标签和行操作）、新增/编辑弹窗、分页。这一张能覆盖契约 §3 的绝大多数条目。

**Q：来源只有两三张设计图，契约填不满怎么办？**
正常。填不满的条目标 `extrapolated` 并把推导规律写进 §8，别硬编。校准阶段这些条目会被重点盯。

**Q：源系统某个组件特别复杂（如可编辑表格），裁决拿不准？**
默认裁「近似」，把差异写清楚拿去确认门。「复刻」要在校准阶段付出成本，确认时想清楚值不值。

**Q：契约里能写业务字段、菜单、示例数据吗？**
不能。契约只放风格事实。业务内容属于校准阶段的单页提示词。

**Q：这个 skill 要跑多久？**
设计目标 1~2 天：取值/归一化半天到一天，黄金页加确认循环半天到一天。确认门返工两三轮是正常的。
