# 输入处理 + 信息充分性检查

> PM 跟客户聊完，手里多是会议纪要、需求文档、界面截图、白板照片。让 AI 直接吃这些，比逼 PM 手写结构化提示词现实得多。本文件规定怎么吃、怎么兜底。

## 1. 接收原则：照单全收 + 充分性兜底

- **鼓励但不强制**低噪声格式（markdown、清晰截图）；PM 给 Word/PDF/微信长图/口述也照收——拦输入会逼 PM 放弃工具。
- 不论什么输入，解析后都要做**充分性检查**（见第 3 节），用协作把"解析不全"显式化，而不是静默出残次品。

## 2. 图片分两类对待

| 图片类型 | 价值 | 处理 |
|---|---|---|
| **已有系统的界面截图** | 金矿——最适合高保真还原 | 走"按已有页面还原"思路：尽量 1:1 复刻布局、字段、列、按钮；对应 erp-prototype-site 手册模板 10 的能力 |
| **手绘草图 / 低清截图 / 糊图 / 扫描件** | 易读错 | 先按能看清的部分理解，**明确提醒 PM 可能失真**，请他用文字补关键点，或换更清晰的图 |

## 2.5 HTML 页面输入（旧原型/第三方系统页面）

当输入为 HTML 文件（旧原型/第三方系统页面）时，**不走"直接生成页面"路径**，而是先执行**翻译层**：

```
旧 HTML 页面 → 外壳剥离 → 结构化提取 → 写成提示词文件 prompts/<编码>-<页面名>.md → 走原有步骤 2~7
```

**核心原则**：旧 HTML 只是需求输入的另一种形态，先把它"翻译"成和文档/口述输入**完全一样的提示词格式**，然后走**完全一样的后续流程**。

翻译层详见 `../../erp-prototype-site/references/html-input-spec.md`（外壳剥离 §2、数量守恒提取 §3、内嵌组件识别 §4、生成后量化自检 §7），此处只规定触发条件和分类：

| 输入类型 | 判断依据 | 处理路径 |
|---|---|---|
| **完整旧原型页** | 有 `<html>`/`<head>`/`<body>`，含外壳（顶栏/侧边栏/页签） | 走翻译层：外壳剥离 → 提取 → 写提示词 |
| **业务内容片段** | 无外壳，只有主内容区 | 跳过剥离，直接提取 → 写提示词 |
| **内嵌组件** | 含 `window.self !== window.top` iframe 检测脚本，或文件名带"内嵌-"前缀 | **不生成独立页面、不注册菜单**，按 html-input-spec.md §4 整合进宿主页面（弹窗/状态 Tab/详情 Tab 内容） |
| **iframe 嵌套页** | 结构明显不完整且无上述信号 | 先询问用户是否能提供独立 HTML；若不能，按片段处理并标记【补位：iframe 结构未知】 |
| **Vue/React 源码** | 文件扩展名是 `.vue`/`.jsx` 或内容含模板语法 | 走模板 10（按已有 Vue 页面还原），不归 HTML 输入处理 |

**提示词产物格式**：与文档拆分产生的 `prompts/*.md` 完全同格式、同位置，后续流程（步骤 2 充分性检查 → 步骤 3 模板建议 → ... → 步骤 6 生成）与文档输入**完全一致**。

## 3. 充分性检查（解析后必做，对接 flow 步骤 2/4）

解析完资料，回吐**两个清单**给 PM：

- ✅ **我读到了**：页面用途 / 查询条件 / 列表列 / 操作按钮 / 跳转或弹窗 / 示例数据线索 …（按实际填）
- ❓ **我没读到 / 不确定**：逐条列出缺口

判断"够不够生成"的最低要素（按页型不同，见各模板的元追问段，但通用至少需要）：
1. 这页是给客户看什么用的（页面用途/类型）
2. 页面中文名
3. 该页型的 1~2 个核心业务要素（列表页=查询条件 + 表格列；表单页=要填哪些字段…）

### 缺口分两类（关键，别把技术问题丢给 PM）

把"没读到"的清单分成两类，**只有第二类才进 flow 步骤 4 的二选一**：

- **技术性缺口 → AI 静默补位，绝不问 PM**：英文编码（**先读 menu.config.js**，按「2位模块+2位子模块+2位序号」推一个不重复的，如承运商归基础信息管理 → `BMCY01`；已有 `BMCY01` 则顺延 `BMCY02`）、菜单分组（按页面性质归到最合适的现有分组）、跳转目标页编码（同样分配一个合规 6 位编码做占位，**不注册进菜单**，并在规格卡补位项标明"占位"；不要用 `-DETAIL` 这类后缀）、图标、示例数据条数（默认 10）。这些 PM 既不懂也给不了，问了只会劝退。补了什么**先写进页面规格卡的"补位项"栏（落盘，给研发看）**，交付时再从该栏复述给 PM（对话清单）。只进对话不落盘 = 违规。
- **业务性缺口 → 走步骤 4 提醒 + 二选一**：状态有哪几种、要不要某个功能按钮、点某处跳到哪个**已有业务页**、关键字段口径等——这些只有 PM/客户知道，值得问。

## 4. 解析质量差时的话术

- 资料明显读取受限（扫描 PDF、糊图、超长无结构文档）时，体谅地说明 + 给替代办法：
  > "这份资料我只读出了一部分（清晰度/格式限制）。要更准的话，你可以：把关键点用几句话打给我，或换一张更清晰的截图。当然你也可以让我先按现在读到的做一版试试。"

## 5. 永不拒绝 / 永不默默硬生成

- 信息不足时**不拒绝**，也**不闷头生成一个错的**。
- 统一走 flow 步骤 4 的**提醒 + 二选一**：(A) 我按常见默认补位并标【补位】先出版 / (B) 你补充后我再开始。
- 选 A 时，把"补了哪些默认值"记下来，交付时作为补位清单交给 PM review。

## 6. 输入分诊（flow 步骤 0 之前，先判断走哪条流程）

在开场立约之前，用可直接观察到的信号（文件数量、材料类型数）判断这次输入属于哪一类：

| 输入形态 | 判断依据 | 走的流程 |
|---|---|---|
| 口述 / 单份文档 / 单张截图，目标做 1 个新页 | 默认场景，材料单一 | 不变，走现有 flow.md 逐页流程 |
| 旧 HTML 页面文件（1~2 个） | 用户提供了 `.html` 文件或粘贴 HTML 代码 | 走翻译层：剥离 → 提取 → 写提示词 → 原有步骤 2~7（见 `../../erp-prototype-site/references/html-input-spec.md`） |
| 旧 HTML 页面目录（多个） | 用户提供了整份旧原型目录 | 触发门二（资源过载），拆分后每页走翻译层 |
| 1~2 个旧原型页面/截图，仅作参照 | 参照物数量少，且用户说的是"做这一页" | 现有"已有页面还原"能力，不用大改 |
| 整份旧原型目录 / 多份文档混合 / "你自己看着办" | 出现目录级输入，或一次给了 ≥3 类不同性质的资料（PRD、编码表、旧原型、其他） | **触发第 7 节两道门 + 第 8 节拆分机制** |

## 6.5 批量执行 · 状态文件检查（新增）

**任何批量输入场景，先检查是否存在执行状态文件**：

1. 检查目标目录是否存在以下文件（按优先级）：
   - `execution-plan.md` → 有 = 已有执行计划，读取后继续
   - `handoff.md` → 有 = 已有交接文档，读取后接续
   - `progress.md` → 有 = 有进度记录，读取后接续
   - `prompts/*.md` → 有 = 已拆分，直接进入生成阶段

2. **状态文件存在时的处理**：
   - **不得重新拆分**：已有 `prompts/` 时，直接读取 `handoff.md` 确认进度，从待办页面继续
   - **不得重新生成已完成页面**：`handoff.md` 中"已完成页面清单"即为不可回滚的产出清单
   - **必须读取进度**：读取 `progress.md` 或 `handoff.md` 中的"待办页面清单"，确认下一批做哪几页

3. **状态文件不存在时的处理**：
   - 首次执行，按正常流程走拆分 → 批次确认 → 逐批生成
   - 拆分完成后立即生成 `execution-plan.md`（总控清单）+ `handoff.md`（初始状态）

> **核心原则**：文件是唯一的可信状态源，对话记忆不可靠。接续时必须从文件重新读取，不得凭记忆"继续"。

## 7. 两道门检查（进入生成前必过，只诊断不拦截）

两道门都在读取任何文件全文之前先做侦察，都只产出诊断报告，不自行决定拒绝执行。符合治理元规则：skill 只负责把信息讲清楚，决定权永远在使用者。

### 7.1 门一：目标兼容性检查

检查用户表达的目标和 `erp-prototype-site` 的产出能力之间是否存在结构性矛盾：

| 用户目标里出现的信号 | 与 skill 能力的矛盾点 | 需要摆出的代价 |
|---|---|---|
| "和旧原型一模一样""像素级还原""完全复刻""换 UI 风格""换组件库""换样式框架" | `erp-prototype-site` 是统一规范生成引擎，不是复刻引擎 | 给 A（默认风格，代价：不像）/ B（提炼风格，代价：只能接近，耗时更长）/ C（直接复用旧文件，代价：丢失 IR 标记，研发不能直接用）三条路 |
| **提供了旧 HTML 页面作为输入** | 意图是"基于旧页面做新原型"而非"要求像素级复刻" | 门一豁免，改为风格确认 A/B/C（见 `../../erp-prototype-site/references/html-input-spec.md` §6）：A=按高端ERP默认风格重新生成 / B=提取旧页面风格写入 theme.css / C=最小改动只注入 IR 标记 |
| "换一套完全不同的 UI 风格/组件库" | 组件库是预置的，替换是工程排期，不是运行时能做到的事 | 说明"临时换风格 = 只能做到接近，正式换风格需要另立项集成" |

命中任意一条，必须在开始生成之前把代价讲清楚、给出选项，用户选哪条都执行。

### 7.2 门二：资源过载检查

**先侦察，不读全量。** 在读取任何文件的完整内容之前，先用轻量操作（`ls`/`find`/`dir`/文件大小）拿到量化信号：

| 信号 | 低风险 | 中风险 | 高风险 |
|---|---|---|---|
| 页面级源文件数量 | ≤5 | 6~15 | >15 |
| 材料类型混杂数（PRD/编码表/旧原型/其他同时出现几类） | 1 类 | 2 类 | ≥3 类 |
| 总文本体量（估算） | 小 | 中 | 明显偏大 |

> 阈值说明：≤5 页是现有逐页流程已验证的安全范围；>15 页来自本次实测（34 个 html + 两万字 PRD 已明显超出安全范围）；6~15 按 2~3 倍安全上限外推。

命中高风险 → 输出诊断报告（第 7.3 节），停在这一步等用户表态。不是默认拒绝——用户一句"别管了，全部做"，立刻执行，同时留痕。

若工具不具备 Shell/文件列举能力（如某些托管环境），降级为直接问用户：
> "为了不让处理过程中途卡死，能不能先告诉我：这次一共涉及几个页面文件？大概多少字/多少张截图？"

### 7.3 诊断报告格式（两道门共用）

不管命中哪道门，统一用这个结构：

> **我看到的输入**：（材料清单 + 数量/体量的具体数字，不说"感觉有点多"）
> **这样输入会带来的问题**：（会不会超出单次任务安全范围 / 会不会和你要的结果本身矛盾）
> **我的建议**：（给可执行的最小修正动作——见第 8 节的拆分，不是一句"请规范输入"）
> **需要你确认的点**：（明确等一个选择，选完立刻执行）

## 8. 拆分流程（命中门二高风险 + 用户同意时触发）

### 8.1 拆分提议

> "要不要我先把这份大文档/整个目录，按页面拆成一份份独立、干净的小提示词，你确认清单没问题，我们再一页页做？"

用户可以跳过这道工序（"别拆了，你自己看着办全部做"），跳过就直接按原始材料做，风险自负，在交付说明里留痕。

### 8.2 单页提示词文件格式

用户同意后，消化原始材料（PRD 定业务口径、编码表定范围、旧原型定视觉参照），产出 N 份单页提示词文件到 `<站点目录>/prompts/<编码>-<页面名>.md`，每份包含：

- 页面编码
- 页面名称
- 页面类型
- 菜单归属
- 查询字段（**必须包含选项值**，不能写"下拉筛选"——要写出具体选项）
- 表格列（**必须包含列宽/对齐方式**，不能写"多列"——要写出具体列名和数据格式）
- 顶部/行操作（**必须包含按钮文案和触发行为**，不能写"操作按钮"——要写出具体按钮名和跳转/弹窗目标）
- 交互要求（**必须包含跳转目标编码或弹窗内容摘要**，不能写"跳转到详情"——要写出符合编码规则的具体编码，如 `BMRK02`）
- 业务说明（**必须写明页面性质**：配置工具页/录入页/查询页/工作台/看板 五选一 + 一句话核心业务流程——实测缺这段会导致"评分卡编辑"被做成打分录入页、"统计报表"被做成报表管理页这类主题跑偏）
- 数量守恒清单（**仅旧 HTML 输入必须**：从源页面数出查询字段数/表格列数/数据行数/页签数/弹窗数/交互点数，生成后逐项核对，见 html-input-spec.md §3.1/§7）

**旧 HTML 输入的额外产物与质量门**：
- 每页除提示词外，剥壳后的业务内容落盘为 `prompts/<编码>-<页面名>.source.html`——生成阶段走**双输入**（提示词做结构索引 + 源文件做细节兜底），见 html-input-spec.md §8
- 生成前必过**提示词质量门**（html-input-spec.md §3.5）：有 JS 运行时跑 `check-prompts.js`，纯对话工具人工过检查表；不达标退回重提取，禁止带病生成——第一次质量差之后的手工调优会脱离 skill 管线，总成本更高

每份干净、自包含——之后喂给 `erp-prototype-site` 的永远是这份干净文件，不是原始的大文档。**长度以信息完整为准，不设行数上限**：文档/口述输入通常 30~80 行；**旧 HTML 输入必须走 html-input-spec.md §3 的数量守恒提取，通常 100~300 行**——实测把 900 行旧原型压成 30 行摘要会导致生成页面丢失 70%~87% 的内容，一句话摘要和"结构摘要"都是不合格输入。

**拆分质量检查（必须执行）**：
1. 检查每份提示词是否包含"【需补位】"标记——这些是 PRD 缺失的信息
2. 统计"【需补位】"数量，写入 `execution-plan.md` 的"风险清单"
3. **命名修饰词检查（新增）**：PRD/旧原型中页面名称含"内嵌""嵌入""弹窗""抽屉""浮层""面板"等词汇时，**先核查实际产物**：
   - **最优先看 iframe 检测信号**：文件内含 `window.self !== window.top` 检测脚本 → 无论是否有完整 `head`/`shell`，都是**嵌入组件**，不生成独立页面、不注册菜单，按 html-input-spec.md §4 判定真实形态（弹窗/状态 Tab/详情 Tab 内容）并整合进宿主页面（实测三个"内嵌-"文件都是完整独立 HTML，仅凭"有完整外壳"会误判为独立页面）
   - 无 iframe 信号、有独立 `.html` 文件 + 完整 `head`/`shell` → 按**普通独立页面**处理，选对应功能模板（清单型→列表查询页、表单型→填表录入页等），正常注册菜单
   - 文件名带“内嵌-”前缀或结构明显只有一个弹窗/Tab 区块 → 同样按内嵌组件处理，不注册菜单，与宿主页面同批处理
   - 不确定宿主页时询问用户；无法确认时才按独立页面处理，并在 `progress.md` 标记 `【页型待确认】`
4. 所有跳转目标编码必须在 34 页清单中存在——不存在的标记 `【跳转目标待确认】`

### 8.3 用户确认 + 批次划分

1. 用户过一遍这些小文件（比读原始大文档轻松得多），勾出本批要做的页数。
2. 批次大小按输入类型定：**文档/口述输入 3-5 页/批**；**旧 HTML 输入 2-3 页/批**（双输入生成 token 消耗大，质量优先于速度），剥壳后 >800 行的复杂页**单页一批**，首批只做 1 页、交付对比报告全 ✓ 后再扩批。每批结束后走交接文档流程（见 flow.md 批次与会话管理）。
3. 消化工作在 `erp-prototype-guide` 这一层做完，不传导给 `erp-prototype-site`。

**批次划分原则（新增）**：
- 同一模块的页面尽量同批处理，保持风格一致
- **命名修饰词检查**：PRD/旧原型中页面名称含“内嵌”“嵌入”“弹窗”“抽屉”“浮层”“面板”等词汇时，优先按 iframe 检测、文件名前缀和组件结构判断；完整 `head`/`shell` 不能覆盖已命中的内嵌信号
- 依赖页面必须先于被依赖页面（如"任务详情"必须在"任务管理列表"之后，因为列表要跳转到详情）
- 首批小批验证：文档/口述输入首批 3 页；**旧 HTML 输入首批只做 1 页**，交付对比报告全 ✓、使用者认可还原度后再扩大批次——静态页面输入时使用者手里有原件，第一页质量就是他对整个 skill 的判断依据
