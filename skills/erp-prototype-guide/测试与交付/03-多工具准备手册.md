# 多工具部署与测试准备手册（Claude → Kimi CLI → Codex → DeepSeek）

> 用途：把 `erp-prototype-guide` 从 Claude Code 扩到其他 3 个工具上验证。
> 本手册写给三类读者：**①你本人（操作者）②被测的 AI 工具（要让它明白自己该干什么）③团队（消除理解误差）**。
> 关联：流程见 `../SKILL.md` 与 `../references/`；测试案例见 `02-统一测试案例(危废承运商).md`。

---

## 0. 先讲原理（所有人必读，消除信息不对称）

- `erp-prototype-guide` 的本质 = **一套 markdown 指令**（`SKILL.md` + `references/flow.md` + `references/input-intake.md` + `references/templates/*`）。
- Claude Code 把它当原生 "skill" 自动加载；**其他工具没有 "skill" 这个概念**，但它们都支持「项目级指令 / 规则 / 自定义提示」。
- 所以"在别的工具上跑 erp-prototype-guide" = **让那个工具加载同一套 markdown，并按里面的流程工作**。
- **单一真源**：这套 markdown 不改；各工具只是用自己的机制把它"挂载"进去。

> 一句话给团队：我们不是给每个工具各写一套，而是**同一套规则，挂到不同工具上**。

---

## 1. 通用准备（你本人，测每个工具前都要做）

1. 把整个 `erp-prototype-guide/` 目录放进**被测项目的工作目录**（让工具能读到这些文件）。
2. 准备好 `02-统一测试案例` 的「客户沟通纪要」文本。
3. 准备一个**空的输出目录**（生成原型站点用）。
4. 录屏就绪。
5. 准备好下面第 2 节的「工作交底文本」（这是让工具进入角色的钥匙）。

---

## 2. 工作交底文本（钥匙 · 工具无关 · 可复制）

> 这段文本是让**任何**工具正确扮演 erp-prototype-guide 的"开场指令"。
> 用法见第 3 节：可作为 Codex 的 `AGENTS.md`、Kimi 的项目指令、DeepSeek 的粘贴文本，或会话开头直接粘贴。

```text
你现在的角色是「产品经理原型引导助手」。请严格按本项目里 erp-prototype-guide 的规则工作：

1. 先阅读这些文件并以它们为准：
   - erp-prototype-guide/SKILL.md
   - erp-prototype-guide/references/flow.md
   - erp-prototype-guide/references/input-intake.md
   - erp-prototype-guide/references/templates/index.md（再按选中的页型读对应模板文件）
   - 生成原型时遵循 erp-prototype-site/SKILL.md 与 erp-prototype-site/references/interaction-ir.md 的规则。

2. 使用者是【非技术产品经理】。全程用业务大白话，绝不向他暴露任何技术标记
   （data-action / data-nav / data-modal / 英文编码等），也不要让他看原始提示词。

3. 按 flow.md 的剧本走：开场先说明"逐页推进"工作方式并征得同意 → 收集需求（可含文档/图片）
   → 充分性检查（回吐"读到/没读到"两清单）→ 自动建议模板并用业务名确认
   → 缺口分类：技术性缺口（编码/分组/跳转占位/数据条数）自己静默补位，只有业务性缺口才"提醒+二选一"
   → 给业务语言"需求确认摘要"让他确认 → 生成【单个】原型页 + 幂等更新 menu.config.js
   → 给补位清单 + 问是否做下一页。

4. 红线：默认逐页推进，PM 坚持批量时提醒风险后照做（见 erp-prototype-guide 治理元规则）；信息不足时永不拒绝、也不默默硬生成（走"提醒+二选一"）；
   生成的页面要真能点（查询/状态页签/弹窗/跳转意图），不得有页面级横向滚动。
5. 若输入量大（>15 个页面文件或混杂 ≥3 类材料），先做两道门诊断（见 input-intake.md 第 7 节），只诊断不拦截，等 PM 表态。
6. 若 PM 提供了旧 HTML 页面文件或粘贴了 HTML 代码，先执行翻译层（剥离外壳 → 提取结构化信息 → 写成提示词文件 prompts/<编码>-<页面名>.md），再走原有步骤 2~7。翻译层产物与文档拆分提示词同格式、同位置。
7. 批量模式每批（3-5 页）结束后更新交接文档（见 handoff-template.md），问 PM 是否开新会话。

9. 若你提供了旧 HTML 页面文件或粘贴了 HTML 代码：
   ① 先执行翻译层：剥离外壳（去掉顶栏/侧边栏/页签）→ 提取业务内容中的字段、列、按钮、弹窗 → 写成提示词文件 prompts/<编码>-<页面名>.md
   ② 提示词格式与文档拆分产物完全一致，后续走同样的充分性检查 → 模板建议 → 缺口处理 → 确认 → 生成流程
   ③ 旧 HTML 只是需求输入的另一种形态，不直接生成页面，而是先"翻译"成标准提示词
```

> 前提：被测工具要能**读到** erp-prototype-guide 这套文件（放进它的工作目录/上下文）。若某工具无法读本地文件，把 `SKILL.md`、`flow.md`、`input-intake.md` 和将用到的模板内容一起贴进去。

## 3. 各工具怎么挂载

### 3.1 Claude Code（基线，顺序第 1）

- **原生 skill 自动加载**：`skills/erp-prototype-guide/` 目录放在 skill 加载路径下，Claude Code 自动识别。
- 完整文件读取 + 命令执行能力，两道门直接走自动侦察。
- **每批改动先在 Claude 上跑 02 统一案例回归**，证明规则本身对，再向下扩。

### 3.2 Kimi CLI（顺序第 2，⚠ 机制需你真机首次确认）

**需要你先确认的点（这是本手册唯一高不确定项，请第一步就验证）：**
- Kimi CLI 是否读取项目根的 `AGENTS.md` 或等价的「项目指令文件」？文件名/位置是什么？
- 它能否在当前工作目录内**读取本地文件**（以便加载 erp-prototype-guide/*）？
- 是否支持「自定义命令/自定义提示」？
- **新增确认项**：能否执行 `node` 命令（决定走 L1 脚本校验还是 L2 填表）。

**确认方法**：查 Kimi CLI 官方文档的"项目配置 / 自定义指令 / AGENTS.md / 上下文文件"章节；或在一个测试目录里放一个 AGENTS.md 写"回答时自称喵助手"，启动 Kimi 看是否生效。

**挂载方案（按确认结果择一）：**
- ✅ 若支持项目指令文件：把第 2 节文本写进它支持的文件（如 `AGENTS.md`），放到含 erp-prototype-guide 的项目根。
- ✅ 若不支持但能读本地文件：**会话开头粘贴第 2 节文本**（万能保底），它据此去读 erp-prototype-guide/*。
- ✅ 若连本地文件都难读：把 SKILL.md + flow.md + input-intake.md + 列表查询页.md 内容连同第 2 节文本一起粘贴。

> 万能保底：无论机制如何，「会话开头粘贴第 2 节文本」这条路一定能走通，不会因为机制没摸清而卡住测试。

### 3.3 Codex（顺序第 3，机制较明确）

- **项目指令**：在被测项目根放 `AGENTS.md`，内容 = 第 2 节文本。Codex 启动会自动加载。
- **可选自定义命令**：在 `~/.codex/prompts/` 放一个 `原型.md`（内容也用第 2 节文本），之后在 Codex 里用 `/原型` 触发。
- 它在项目目录内可读本地文件，能加载 erp-prototype-guide/*。
- 有命令执行能力，校验走 L1 脚本。
- 注意：Codex 对中文长指令的服从度需在首测时重点观察"不暴露技术词"红线（03 手册 §5 列的高发差异第一条）。
- 注：以你实际的 Codex 版本/文档为准；若 `AGENTS.md` 行为有出入，退回"会话开头粘贴第 2 节文本"。

### 3.4 DeepSeek（顺序第 4，全新，纯对话形态）

DeepSeek 网页/App 无文件系统，是四工具里约束最强的，单独设计工作形态：

1. **挂载（L3）**：会话开头粘贴「交底文本 2.0」+ 以下文件内容（按此顺序，够用为止）：
   `erp-prototype-guide/SKILL.md` → `flow.md` → `input-intake.md` → 选中页型的模板 → `erp-prototype-site/SKILL.md` 的 Hard Rules 段 + `page-template.html` 全文。
   为降低粘贴负担，建议实施时在 `erp-prototype-guide/` 下**新增一份 `DeepSeek挂载包.md`**（把上述内容按序拼好 + 交底文本置顶），操作者一次复制即可。该文件由脚本或手工从真源拼接生成，**注明"生成物，勿直接编辑"**，避免破坏单一真源。
2. **产物交付**：无法写盘 → 约定 DeepSeek 每页输出**两个完整代码块**：① 页面 HTML 全文（首行注释给出目标路径 `pages/<分组>/<页面名>.html`）② `menu.config.js` 全文（非 diff，全文覆盖，避免操作者手工合并出错）。操作者按注释路径保存。此约定写进交底文本 2.0 的 DeepSeek 附加段。
3. **校验**：走 03 §3 的 L2 自检填表（强制证据摘录）。
4. **图片输入**：首测确认其截图理解精度；不达标时按 `input-intake.md` §2 的"糊图"路径处理（提醒失真 + 请 PM 文字补关键点），不必为 DeepSeek 单独降低还原标准。
5. **menu.config.js 状态同步**：DeepSeek 看不到磁盘上的最新 menu，连续做多页时靠上一轮输出的全文延续；**新会话续做时，操作者须把当前 menu.config.js 全文粘给它**（此提醒写进挂载包开头）——否则编码顺延与查重（02 §1.2 第 1 步）会失效，这是 DeepSeek 形态下 P2 问题的唯一复发通道，必须显式堵住。

### 3.5 本次新增机制的按工具注意事项

> 以下为 05 号设计方案中新增机制（输入分诊、两道门、拆分、批次交接）在各工具上的适配要点，写进此手册统一管理。

**Claude Code（基线）**：
- 两道门的侦察直接用 Bash/Glob 等工具跑出精确数字，无需降级。
- 交接文档天然跨会话生效（文件 I/O 可靠），是完整验证基线。

**Kimi**：
- 交接文档必须是独立文件而非依赖对话记忆——Kimi 实测中出现过异常退出，文件是唯一可靠的状态载体。
- 工作交底文本里两道门的规则要写成简短祈使句中文，不要长段抽象描述（参考 `原型skill使用与改进建议.txt`："Kimi 对中文长规则黏着度更好"）。
- Shell 能力待确认：若无法跑 `ls`/`find`，两道门侦察降级为直接问用户要数字。

**Codex**：
- 两道门的侦察可直接用 shell 命令，机制明确，风险最低。
- `AGENTS.md` 里已包含新增机制的编排段落（见工具无关编排提示词），无需额外操作。

**DeepSeek**：
- 先完成第 3.4 节的能力确认，再决定走哪条降级路径（见下）。

### 3.6 降级路径：工具没有可靠的文件/Shell 能力时

若目标工具经确认不具备 Shell/文件列举能力（典型如 DeepSeek 在某些托管环境下，或 Kimi 某些版本），两道门的资源过载检查退化：

> "为了不让处理过程中途卡死，能不能先告诉我：这次一共涉及几个页面文件？大概多少字/多少张截图？"

不要凭模型自己数上下文里贴的内容——因为凭感觉容易数错。这条降级路径是每个新工具接入前必须先确认的第一件事：**先探测工具的文件/Shell 能力，再决定两道门的侦察方式走"自动数"还是"问用户"。**

## 4. 每个工具的测试（统一用 02 案例）

1. 按本工具的挂载方案让它进入角色。
2. 走 `02-统一测试案例` 的输入与标准答案。
3. 用 02 的「验收点」打分，并填 02 末尾的「横向对比记录表」一行。
4. **校验层**：有命令执行能力的工具（Claude/Kimi/Codex）跑 `check-site.js` 看是否全 ✓；无命令的（DeepSeek）看交付末尾是否出现全 ✓ 自检表，抽查 3 条证据摘录与生成文件一致。
5. **Kimi 这一轮按要求录屏**（和 Claude 同案例，便于团队对比）。

---

## 5. 出问题怎么记录（反馈给开发）

每个问题记：**哪个工具 / 第几步 / 你说了什么 / 它实际怎么做 / 期望怎么做 / 录屏时间点**。
特别关注五类高发差异：
- **没遵守"不暴露技术词"**（弱模型容易把 data-action 等吐出来）。
- **选错模板**（AI 选型不准）。
- **该静默补的技术缺口反而来问你**（缺口分类没执行好）。
- **编码格式/顺延错误**（6 位 2+2+2 结构不对、与 menu 重复、未先读 menu.config.js）。
- **补位项栏缺失或空置**（规格卡 2.0 三栏不全、补位项为空或未写"无"）。

这些大多能通过强化第 2 节文本或对应模板措辞来修，少数是工具能力限制。
