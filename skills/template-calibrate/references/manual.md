# 模板校准 skill（template-calibrate）— 使用手册

> 给使用者看的操作手册。重点是**大量可拷贝的提示词案例**——复制下来，改【】里的内容就能用。
> 这是一个跨多次会话的长流程 skill：所有状态在 `xdocs/模板校准/<pack-id>/` 的台账文件里，任何一次会话中断都能靠文件恢复现场。

---

## 1. 这个 skill 帮你做什么

输入一个风格包草案（「模板提炼」skill 的产出），它：

1. **第0批**：实例化新风格的三件套家族（换风格绑定层+打版本戳），补出 Vue 侧样式投影和 Vue 侧黄金页，跑双栈平行校验；
2. **第1~N批**：按页面类型分批跑真实页面（单页提示词→原型 HTML→转 Vue→mock 验收），**每个发现的问题强制三分类**，可泛化的走 7 步回灌协议改进模板；
3. **收敛定版**：类型覆盖完成 + 连续一批零新规则 → 风格包冻结 v1.0，家族发布。

它复刻的是石油焦掺配模型 18 页批量优化被验证过的路径，把当时靠经验维持的纪律（质量闸门、台账、change-log、回灌）变成强制流程。

另一个用途：**现有家族回炉**——哪天发现原型和 Vue 产出又漂移了，用它对既有风格包跑一个迷你批次拉齐。

---

## 2. 开工前提

- 风格包草案就位：`style-packs/<pack-id>/` 有 contract.md（≥v0.1）、prototype/ 全套、已确认的黄金页、parity-checklist.md；
- 分层后的三件套脚手架可用（风格无关层可复用）；
- 校准素材：待做页面清单及业务依据（旧原型/设计图/文字描述）。

---

## 3. 提示词案例库（复制即用）

### 案例 1：全新开工

```
用「模板校准」skill 开工，校准风格包【cnal-green】。

输入：
- 风格包：kimi-skill-project/style-packs/cnal-green/（契约 v0.1，黄金页已确认）
- 家族目标位置：【kimi-skill-project/skills-cnal/】
- 校准素材：【xdocs/资料/新系统页面清单.md，共约 15 页，业务依据是设计图+文字描述】

先做三件事，做完停下等我确认：
1. 建工作目录 xdocs/模板校准/cnal-green/，从模板铺好六个状态文件；
2. 跑版本戳校验（首次开工应该全空，记录基线）;
3. 盘点素材页面、归类到 7 类页面原型，出批次计划草案（每批 3~5 页，
   标准列表页在前，配置矩阵/向导压后）。
```

### 案例 2：接手中断现场（新会话第一条消息）

```
用「模板校准」skill 接手 xdocs/模板校准/【cnal-green】/。
按接手说明的顺序来：先读执行台账对齐状态，跑版本戳校验，
检查待议清单有没有等我裁决的行，然后告诉我"下一步动作"是什么再继续。
```

### 案例 3：执行第0批

```
批次计划我确认了，执行第0批：
1. 从分层脚手架复制实例化【skills-cnal】家族，风格绑定层全部换成
   cnal-green 派生副本，每个派生文件头部打戳 cnal-green@0.1.0，
   三件套 SKILL.md 记录依赖版本；
2. 写 Vue 侧投影：vue/prototype-parity.scss + variables.scss，
   数值全部来自 contract.md，写完先跑一遍 parity-checklist（L1）；
3. 用新家族的 prototype-to-vue3 把黄金页转成 Vue 黄金页，serve:mock 跑通，
   执行 Runtime Acceptance（直开路由、刷新不弹404、主流程按钮无报错、控制台干净）；
4. L2：双栈黄金页各截一张整页图给我并排看；浏览器工具不可用就告诉我，
   我自己开两个窗口目视，台账记"部分通过"；
5. 台账第0批各项置状态，停在"执行者确认门"等我。
```

### 案例 4：批次执行（最常用的日常提示词）

```
执行第【1】批（标准列表页：【NMGL01 用户管理、NMGL02 角色管理、NMGL03 参数维护】）。

每页按单页流水线走：
1. 用 assets/单页提示词-模板.md 生成 prompts/<编码>-<名称>.md，
   业务内容从【设计图/旧原型路径】提取，样例数据要真实感、覆盖所有状态；
2. 提示词给我过目后再生成 HTML 到 html/ 目录，逐项过"生成后检查点"；
3. HTML 我确认后转 Vue，serve:mock 自检 + Runtime Acceptance；
4. 每页台账实时更新状态。

纪律重申：过程中发现的任何问题（哪怕看起来只是小样式）都要三分类报给我，
不许顺手默默修掉；可泛化的先挂待议，批次末集中走回灌。
```

### 案例 5：问题三分类 + 回灌实战（附真实范例）

```
第【1】批跑完了，把本批发现的问题按三分类列表报我：
| 问题 | 现象 | 你建议的分类 | 理由 |
我裁决后：
- "仅本页修复"的直接修，台账备注；
- "可泛化→回灌"的逐条走 7 步协议，每条给我看 CHANGELOG 节草稿再落；
- 拿不准的进待议清单。
```

回灌协议 7 步的执行范例（以石油焦时期真实问题「窄表被全局宽表 min-width 撑出横向滚动条」为例，供对照）：

| 步 | 动作 | 该例中的落点 |
| --- | --- | --- |
| 1 | 改契约升版 | contract §3 新增 fit-table 行为约定，v0.2→v0.3 |
| 2 | 同步双侧投影 | app.css 加 fit-table 规则；prototype-parity.scss 加 `.prototype-table.fit-table` |
| 3 | 跑 L1 | parity-checklist 追加 fit-table 行，全项核对 |
| 4 | 同步派生副本+版本戳 | 三件套内 app.css/parity.scss 副本更新，戳升 @0.3.0 |
| 5 | 规则入 skill | prototype-site 硬规则加"窄表禁无意义横向滚动"；prototype-to-vue3 加"窄表不传 scroll.x"+自检项 |
| 6 | CHANGELOG 追加一节 | 背景（哪页暴露、客户反馈）/关键规则/已改文件/下游影响 |
| 7 | 已完成页返工判定 | 排查已通过页面里的窄表，登记返工行 |

### 案例 6：批次验收

```
第【1】批三分类处置完毕，做批次验收：
1. cd 【家族框架目录】 && npm test && npm run build，结果记验收记录；
2. 本批有回灌 → 跑全量 L1；
3. L2 抽查【NMGL02】双栈截图（不可用则我目视）；
4. 验收记录按模板追加"批次1"一节，待议清单确认无未裁决行；
5. 更新接手说明的快照和交接记录，然后停下——批次结论我来签。
```

### 案例 7：收敛判定与定版

```
第【4】批零回灌，做收敛判定：
1. 从台账拉类型覆盖表和批次规则产出记录，两条判据逐条核对给我看
   （不存在的类型确认"不适用"理由成立）；
2. 判据满足且我确认后执行定版：契约 §0 → v1.0 冻结、全部版本戳同步、
   CHANGELOG 写定版节、最终跑一遍 L1、黄金页 Runtime Acceptance；
3. 接手说明置"已定版"，写清维护规则：以后任何风格改动走回灌协议、升 minor 版本。
```

### 案例 8：现有家族回炉（漂移修复）

```
用「模板校准」skill 给现有家族做回炉，不是新风格：
风格包【lxlt-blue】@1.0.0，家族【skills/prototype-*三件套】。
现象：【用户反馈 Vue 页面表头颜色和原型不一致】。
1. 跑版本戳校验和全量 L1，把所有不一致项列出来；
2. 逐项判定漂移方向：谁改过了（对照 CHANGELOG，没有记录的改动就是越权改动）；
3. 以契约为准回灌拉齐（合理的越权改动先补契约变更，再走 7 步；
   不合理的直接从契约重新派生覆盖）；
4. 跑一个 1~2 页迷你批次验证拉齐效果。
```

### 案例 9：返工执行

```
处理台账"返工登记"里 CHANGELOG 第【2026-XX-XX 窄表fit-table】节触发的返工：
影响页面【NMGL01、NMGL03】。逐页改（原型 HTML 和 Vue 同步改），
过各自检查点后台账返工行置"已通过"，页面行备注返工完成。
```

## 5. 常见问题

**Q：一批里问题太多，回灌是不是太重了？**
头两批回灌多是正常的（石油焦第一轮五条体验规则全是前期爆的）。7 步一步都不能省——漂移的代价已经付过一次；越往后批次越干净，最后一批零回灌就是收敛信号。

**Q：L2 截图一直做不了怎么办？**
按软门设计走：人工目视 + 台账记「部分通过」，不阻塞。但定版前建议至少对黄金页做一次认真的人工双栈对比，这是数值核对兜不住的最后一道。

**Q：批次中途我想改个颜色，能直接改 parity.scss 吗？**
不能，这是第一铁律。哪怕一个色值也走回灌：改契约→升版→同步双侧→L1→同步副本→CHANGELOG。直接改副本 = 下次从契约派生时被覆盖 + 双侧漂移。

**Q：某页做完我不满意但说不清哪里不对？**
台账置「需返工」，问题进待议清单写"观感不达预期，待拆解"。下次会话拿它和黄金页/来源并排比对，把"说不清"拆成可三分类的具体条目。

**Q：中途换了台电脑/换了会话，怎么保证不乱？**
只要台账是实时更新的就乱不了。新会话永远用案例 2 的接手提示词开场；skill 的硬规则要求先对台账再干活。

**Q：页面清单后来新增了怎么办？**
新增页归类到所属类型批次；若已收敛定版后新增，按正常业务开发走新家族的三件套即可——校准是给模板定型的，不是每页都要过校准。
