# ERP Prototype To Vue (Vue3 edition)

Convert static prototype business pages into the Vue3 demo framework generated by `erp-prototype-vue3-framework` (**Vue 3.4 + Ant Design Vue 4**). Static paths stay Chinese; Vue paths are derived from the English `data-page` code.

## Resources

- Read `references/psm-ui-profile.md` before converting naming, menus, styles, or component patterns.
- Read `references/interaction-ir.md` before translating `data-action`, `data-modal`, or `data-nav`.
- Read `references/visual-parity.md` before converting page layout or styles.
- Read `references/manual.md` only for extended examples or troubleshooting.

## Hard Rules

- Vue files must be English-coded: `src/views/<模块>/<子模块>/<编码>.vue`.
- API files must be `src/views/<模块>/<子模块>/<编码小写>Api.js`.
- Derive `<模块>/<子模块>` from the code's letter part (the code minus its trailing 2 digits): module = first 2 letters, submodule = the remaining letters. Standard 6-char codes: `BMWL01 -> BM/WL`, `BMSP04 -> BM/SP`. **旧编码兼容**（erp-prototype-site 允许沿用 `^[A-Z]{2,6}[0-9]{2}$` 的旧编码）：`FXGLQD01 -> FX/GLQD`, `RWJSC01 -> RW/JSC`. Legacy codes are valid input — do not rename them to 6 chars during conversion (renaming breaks `data-nav` targets and the correspondence with the old system).
- `menu.js` must satisfy `formEname == 编码 == .vue filename` and `formLoadPath == /views/<模块>/<子模块>/<编码>`.
- Chinese page names go only into display fields such as `formCname`, `groupCname`, and `fullTitle`.
- Use mock-only API files; do not connect to a backend.
- Emit **Vue 3 + Ant Design Vue 4** idioms (see `references/psm-ui-profile.md` §6/§8): `v-model:value` on inputs/selects; `<a-form>` + `<a-form-item name>` (no `a-form-model`); table single `#bodyCell="{ column, text, record, index }"` slot with object-arg `customRender`; `v-model:open` modals; `:deep()` deep styles; button icons via `<template #icon>`; `<a-icon type="...">` still works through the framework's `AIcon.js` compat layer.
- Converted `.vue` files must not contain `EFGrid`, `EFPage`, `XrBaseView`, `OPERATE_TYPE`, `#0052d9`, `location.href`, `postMessage`, `PR.`, nor any Vue2/AntD1 leftovers: `a-form-model`, `slot-scope`, `scopedSlots`, `::v-deep`, `:visible=` on `a-modal`, or bare `v-model=` on form controls.
- `formLoadPath == /views/<模块>/<子模块>/<编码>`; the framework adapter (`mock.js` `loadView`) appends `.vue` → `import('@/views/<模块>/<子模块>/<编码>.vue')`, so the `.vue` file must exist at that path.
- Translate `data-nav="<编码>"` to `this.$router.push('/<编码>')`.
- **No raw DOM access in the output.** Prototype behaviors that scroll/highlight/focus a region (`this.$el.querySelector(...)`, `querySelectorAll`, `getElementById`) must be converted to a `ref` + the equivalent method (e.g. `ref="calcCard"` then `this.$refs.calcCard?.scrollIntoView({ behavior: 'smooth' })`). In Vue 3 `this.$el` of a multi-root/fragment component has no `querySelector`, so a carried-over `$el.querySelector` throws `TypeError: this.$el.querySelector is not a function` on first interaction.
- **Every `$emit` needs a matching `emits`.** If a converted page introduces a child component that `$emit('xxx')`s and the parent listens with `@xxx`, the child must declare `emits: ['xxx']` — otherwise the listener falls through to the child's root node (especially when the root is an AntD wrapper), firing twice and passing the wrong payload. Flag any `$emit('xxx')` without a declared `emits` as a high-priority runtime risk.
- **Port site-level style extensions, or fidelity silently degrades.** erp-prototype-site allows appending new component classes to the site's `assets/app.css` inside a `/* === 站点扩展区（只增不改） === */` block (drawers, switches, radar charts, selected-state cards, …) and skin overrides in `assets/theme.css`. `prototype-parity.scss` in the Vue framework only covers the standard `prototype-*` classes. Before converting a page, grep the page's class names against the site's app.css 扩展区 and theme.css: every extension class the page uses must be reproduced on the Vue side — page-scoped styles for one-off classes, or appended to the framework's `prototype-parity.scss` inside its own marked extension block (append-only, never modify existing rules) when shared by multiple pages. theme.css skin overrides (colors/fonts/radius/shadows) port the same way. A converted page that references extension classes with no Vue-side definition fails visual parity even if the build is green.
- **High fidelity is required.** Reproduce the prototype layout 1:1 — do not redesign with Ant Design Vue defaults. Keep the prototype section order, stat cards, query grid, card headers, status tabs, batch bar, table density, action placement, and the modal's column layout, using the shared `prototype-*` classes (mechanical class rename per `references/psm-ui-profile.md` §6 table). Mirror the gold-standard sample `src/views/BM/SP/BMSP01.vue` in the framework. Concretely: convert `query-grid` to `prototype-query-grid` (4-col grid, **not** `a-form layout="inline"`); keep `status-tabs`; convert a 2-column `modal-body` to `a-form layout="vertical"` wrapping `.prototype-modal-form` (**not** a single-column horizontal form); map native controls to `a-input`/`a-select`/`a-range-picker` and the all-rows checkbox to the table `row-selection`.

## Single-Page Workflow

1. Read the static page and identify `data-page`, parent folder as `groupCname`, page filename as `formCname`, and the matching `menu.config.js` row.
2. Derive Vue output paths from the code.
3. Convert page content to Ant Design Vue while preserving visual structure: keep `prototype-page`, `prototype-card`, explicit card heads, `prototype-query-grid`, status tabs, table wrappers, and modal form layout. Use AntD for behavior but override its default spacing through the shared parity layer.
4. Move `MOCK_DATA` into the API file and implement query/filter/pagination methods returning `{ rows, count }`.
5. Add or update one `menuRows` entry in `src/mock/data/menu.js` by `formEname`.
6. Run the self-check and build when dependencies are available.

## Whole-Site Workflow

1. Read `menu.config.js` from the prototype site, then read the pages it registers. **Convert only pages registered in `menu.config.js`.** Files under `pages/` that are not registered — typically `内嵌-*` embedded-component leftovers kept for reference (their content was already integrated into a host page as a modal/status tab/detail tab per erp-prototype-site's html-input-spec.md §4) — must be skipped, not converted into standalone Vue pages or menu rows.
2. Convert each page with the single-page workflow. Existing `.vue` and API files are overwritten to match the latest prototype; unrelated pages are left alone.
3. Merge all site menu rows into `src/mock/data/menu.js` idempotently by `formEname`.
4. Report added/updated Vue pages, API files, and menu rows.

## Self-Check

- The set of `menu.js` `formEname` rows for converted pages equals the static site `menu.config.js` code set. Unregistered `pages/` files (e.g. `内嵌-*` reference leftovers) produced no Vue page and no menu row.
- Legacy codes (`^[A-Z]{2,6}[0-9]{2}$`) kept their original spelling; module/submodule derived from the letter part (first 2 letters / remaining letters).
- Every extension class the page uses (from the site's app.css 站点扩展区 or theme.css) has a Vue-side definition — page-scoped or in the parity layer's extension block; no class referenced by the template resolves to nothing.
- Every converted code has a real `.vue`, API file, and matching `formLoadPath`.
- No converted Vue path or filename contains Chinese.
- All actions have methods, modals open and close, and navigation uses `$router.push`.
- Vue3/AntD4 idioms only: `v-model:value`, `<a-form-item name>`, table `#bodyCell`, `v-model:open`, `:deep()`; no `a-form-model`/`slot-scope`/`scopedSlots`/`::v-deep`/`:visible=`/bare `v-model=` leftovers.
- Visual fidelity checklist from `references/visual-parity.md` passes: card/header/table/query/modal/status tab structure remains equivalent to the static prototype.
- No page-level horizontal scroll (`document.scrollingElement.scrollWidth <= clientWidth`); narrow tables use `fit-table` (no `:scroll="{ x }"`); paired cards in a `prototype-cols` row are equal-height; final actions use a fixed `fixed-bottom-bar` with `has-fixed-actions` on the page root.
- No raw DOM access carried over: no `this.$el.querySelector`/`querySelectorAll`/`getElementById` in the output; scroll/highlight/focus behaviors use `ref` instead.
- No `$emit('xxx')` without a matching `emits: ['xxx']` declaration on the same component.
- `npm run build` succeeds when the framework dependencies are installed.

## Runtime Acceptance (beyond build + SPA clicks)

A page that builds and navigates via in-app clicks can still break on real use. Before declaring a page done, also verify:

- Direct-URL open of the page route (`#/<编码>`) renders the page, and a full-page refresh on it stays put (does not bounce to `/404`).
- The key interactions of the page run without a runtime error / webpack dev overlay — click the primary-flow buttons (search/save/the page's signature action such as scroll-to/preview/试算), not just open the page.
- Console stays clean on those interactions (no `$el.querySelector is not a function`, no double-fired handlers).
