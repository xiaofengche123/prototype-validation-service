# 向导访谈剧本

> 向导按当前阶段选用对应剧本节。规则：一次一问（最多 2-3 个强相关的并列问）、每问带推荐默认值和一句"为什么问这个"、答案由向导落到引擎产物里，用户永远不碰模板和提示词。
> 问题措辞可以按对话语境调整，但**信息点不能少问、也不能多问**（文件系统能查到的不许问）。

---

## §1 新开工访谈（无风格包、无校准目录）

### 1.1 定来源（决定走 A/B/C 哪条路径）

> 问法上不要让用户背"路径A/B/C"，问他手里有什么。

**Q1**：这次的新样式，你手里有什么？（可多选）
- ① 一个能跑起来的旧系统（浏览器能打开看页面）
- ② UI 设计图（图片/设计稿导出），可能还带文字说明或一份别的工具生成的 HTML
- ③ 一批现成的、看着满意的静态原型页面
- 为什么问：决定提炼路径和风格真相认定。

**路由**：只有① → 路径A；只有② → 路径B；只有③ → 路径C；多选 → 追问 Q1a。

**Q1a**（多来源时）：哪个代表"新样式该长的样子"？其他来源只当业务参考。
- 推荐默认：有设计图选设计图（客户认的是它）。
- 落点：契约 §0 风格真相裁定。

### 1.2 定基本盘

**Q2**：给这套风格起个英文代号（pack-id）？
- 向导先按业务给 1~2 个建议（如 `audit-dark`、`cnal-green`），用户可直接选。
- 为什么问：目录名/版本戳/工作目录都用它，定了不改。

**Q3**：来源资产放在哪？
- ① 让向导在常见位置找（xdocs/资料/ 等）并列出候选让用户勾认
- ② 用户直接说路径/地址（路径A 是系统地址+登录方式）
- 推荐默认：①。

**Q4**（仅路径A）：向导先扫描源系统的页面，自动按 7 类原型归类并列出分布表，然后问用户：这个归类对不对？有没有缺的页面类型？
- 落点：sources/ 截图归档 + 后续批次计划的类型覆盖表。
- 用户只需确认或修正，不需要自己口头报页面和类型。

**Q5**（仅路径B）：设计图画了哪几张页面？带的 HTML 是谁做的、能信到什么程度？
- 向导提醒：HTML 只当"页面上有什么"的参考，颜色间距一律以图为准（引擎硬规则，提前说明防误会）。

**Q6**（仅路径C）：给定页面里哪一张信息最全（有查询区、表格、状态标签、弹窗的那种）？拿不准就让向导看一遍目录后推荐。
- 落点：黄金页候选。

### 1.3 执行与门

访谈够了就出「我将这样做」小结（3~5 句大白话），用户点头后向导驱动 template-distill。到黄金页确认门时：

**门话术**：黄金页做好了，文件在 <路径>，双击打开。我念一条你看一条：
1. 主色和状态标签颜色，和来源对得上吗？
2. 卡片标题栏的高度和样式，看着一致吗？
3. 查询区的排布和控件高矮呢？
4. 表格的表头颜色、行高、鼠标悬停效果呢？
5. 弹窗打开看下布局；按钮都点一遍，有没有点了没反应的？
6. 页面往右拖，有没有整页横向滚动条？
- 每条收 通过/不对（说哪里不对），向导落确认卡；有"不对"就返工循环，改完再走门。

## §2 校准开工访谈（风格包 v0.x 就绪、无校准目录）

**Q1**：新家族三件套放哪个目录？
- 推荐默认：`kimi-skill-project/skills/prototype-site-<pack-id>/`（放在 skills/ 下，后续需 junction 到 `.claude/skills/` 注册）。
- 向导自动提醒：产出后需要在 `.claude/skills/` 下建目录联结才能被系统识别。

**Q2**：这轮要做的页面清单从哪来？
- ① 有现成清单文件（说路径）
- ② 从来源资产里盘（向导去数设计图/旧系统/给定原型有哪些页）
- ③ 现在口头报几个先跑着
- 推荐默认：②。

**Q3**：清单盘出来后向导按 7 类归类并标风险，出批次计划草案给用户看。用户只需回答：批次顺序有没有要调的？有没有哪页特别重要要提前/特别怪要押后？

**第0批执行后·双栈黄金页门话术**：两边各开一个：原型版 <路径> 和 Vue 版 http://localhost:xxxx/#/<编码>。并排放，逐条看：颜色/卡片头/查询区/表格/弹窗/按钮尺寸——两边看着是"同一张页"吗？哪里不像说哪里。
- 结果落台账第0批确认门行；通过则风格包升 v0.2。

## §3 续跑（校准目录已存在）

> 不访谈。动作顺序固定：

1. 读 接手说明 + 执行台账，输出三句话现状：「上次做到 X；有 N 个问题等你裁决；下一步是 Y」。
2. 有待议行 → 先逐条给用户裁决（每条给建议分类和理由，用户选：本页修 / 回灌 / 再放放）。
3. 无阻塞 → 一句话确认下一步（"继续给 XX 页生成原型，好吗？"）→ 干活。
4. 批次末问题三分类表、批次验收结论，都是给用户看结果做选择，不让用户写任何东西。

**问题三分类话术**（每批末）：这批发现 N 个问题，我建议这样分——表格列出：问题｜现象｜建议分类｜理由。你逐行看：同意打勾，不同意说改成哪类。回灌的我会走完整 7 步并把变更记录给你过目。

## §4 回炉（风格包已 v1.0，用户反馈漂移/不一致）

**Q1**：不一致的现象是什么？在哪个页面看到的？（用户用大白话描述即可，如"Vue 的表头颜色和原型不一样"）

**动作**：版本戳校验 + 全量 L1 → 列出所有不一致项和"谁被改过"（对照 CHANGELOG，无记录的改动即越权改动）→ 每项给用户二选一：
- ① 这个改动是对的，补进契约（走回灌 7 步）
- ② 这个改动不该有，从契约重新派生覆盖掉
→ 拉齐后跑 1~2 页迷你批次验证，报结果。

---

## §5 通用话术规范

- 永远先说人话结论，再给细节；文件路径给全，方便用户点开。
- 每个自动执行段落跑完，用 3 句以内说清：做了什么、产物在哪、下一步是什么。
- 用户答非所问或提出新需求时：先判断是否偏出旅程（只想做张原型页 → 指去 prototype-site），不硬套向导流程。
- 禁语：不说"请按照 XX 模板填写"、不说"路径A/B/C"（内部术语）、不甩引擎 skill 的英文规则原文。
