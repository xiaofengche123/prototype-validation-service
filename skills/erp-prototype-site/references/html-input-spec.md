# HTML 页面输入处理规范（翻译层）

> 当输入为旧 HTML 原型/第三方系统页面时，本文件是翻译层的唯一执行规范。
> 背景：实测中"旧 HTML → 一句话摘要 → 生成"导致全站平均只还原 13% 内容（14,681 行 → 1,872 行），
> 根因是信息压缩比高达 30:1。本规范的核心目标是把压缩比降到 3:1 以内。

## §1 触发与分类

| 输入类型 | 判断依据 | 处理路径 |
|---|---|---|
| 完整旧原型页 | 有 `<html>`/`<head>`/`<body>`，含外壳（顶栏/侧边栏/页签） | §2 剥离 → §3 提取 → 写提示词 |
| 业务内容片段 | 无外壳，只有主内容区 | 跳过剥离，直接 §3 提取 |
| 内嵌组件 | 命中 §4 的任一识别信号 | 走 §4，**不生成独立页面** |
| Vue/React 源码 | `.vue`/`.jsx` 或含模板语法 | 走"按已有 Vue 页面还原"模板，不归本规范 |

**硬性要求：必须完整读取旧 HTML 全文后再提取。** 禁止只读文件开头或凭文件名/标题猜测页面内容——实测中"统计报表"页因此被生成为完全无关的"报表管理"页（业务语义偏差）。

## §2 外壳剥离

- 移除：顶栏（top-header/header/navbar）、侧边栏（sidebar/menu）、页签栏（page-tabs）、面包屑中与外壳重复的部分、`<html>`/`<head>` 中的外壳级引用。
- 保留：主内容区的**全部**业务结构，包括查询区、统计卡、标签区、表格、弹窗、抽屉、分页器、底部操作栏、页面级 `<style>` 与 `<script>`（作为 §3 的提取来源，不直接照搬）。
- 剥离外壳 ≠ 压缩内容。剥离后业务内容的信息量必须与原页面等价。

## §3 结构化提取与映射规则（核心）

### 3.1 数量守恒清单（必做，写入提示词文件头部）

从旧 HTML 中**数出**以下数字并记录，生成后逐项核对（见 §7）：

```
### 数量守恒（源页面实测）
- 源文件行数：N
- 查询字段数：N（逐个列出）
- 表格数：N，每表列数：N/N/...（逐列列出列名）
- 数据行数：每表 N 行
- 状态页签/Tab 数：N（逐个列出，含计数徽标）
- 统计卡数：N（逐个列出指标名）
- 弹窗/抽屉数：N（逐个列出名称与内含字段）
- 操作按钮数：顶部 N + 行内 N（逐个列出文案）
- 图表数：N（类型：雷达/折线/饼/柱）
- 交互点：展开收起/级联/筛选联动/步骤切换等，逐条列出
```

### 3.2 元素映射规则

| 旧 HTML 元素 | 提取为 | 备注 |
|---|---|---|
| `<th>` / 表头单元格 | 表格列定义（列名/宽度/对齐/数据格式） | 列数必须一致，禁止合并省略 |
| 表单控件（input/select/textarea） | 查询字段或表单字段（名称/类型/**具体选项值**/默认值/必填） | select 的 option 逐个抄录，禁止写"下拉筛选" |
| `onclick="fn()"` / `addEventListener` | 交互点清单（触发元素/行为/目标） | 转写为 data-action/data-nav/data-modal 语义 |
| `.modal` / `.drawer` / overlay 结构 | 弹窗/抽屉定义（名称/内含字段/触发按钮/底部按钮） | |
| Tab/页签结构 | 状态页签定义（名称/计数/**每个页签下的数据样例**） | 每个页签切换后必须有非空数据 |
| `<svg>` / `<canvas>` / 图表容器 | 图表规格（类型/数据维度/样式要求） | |
| 写死的表格行数据 | mock 数据样例（保留原始数据值，条数 ≥ 源页面） | |
| 页面级 `<style>` | 视觉意图描述（卡片选中态/hover/动画等），供 theme.css 或 app.css 扩展区实现 | 不照搬 CSS，但视觉效果必须记录 |

### 3.3 业务语义段（必写，防止主题跑偏）

提示词文件必须包含：

```
### 业务说明
- 页面性质：配置工具页 / 录入页 / 查询页 / 工作台 / 看板（五选一，从旧页面实际功能判断）
- 核心业务流程：一句话说清用户在这页做什么
- 易混淆点：如"评分卡编辑"是配置评分模板，不是给供应商打分
```

实测两类语义偏差都源于缺这段：①"评分卡编辑"被理解为打分录入页；②"统计报表"被生成为报表管理页。

### 3.4 提示词文件产物（双文件落盘）

- 提示词落盘为 `prompts/<编码>-<页面名>.md`，格式 = 数量守恒清单 + 逐区块明细（查询区/表格区/交互点/图表/业务说明）。
- **剥壳源文件同时落盘**为 `prompts/<编码>-<页面名>.source.html`：外壳剥离后的完整业务内容（含页面级 style/script 原文）。它是生成阶段的第二输入（见 §8），也是交付对比的基准。
- **长度以信息完整为准**：旧 HTML 输入的提示词通常 100~300 行；剥壳后源文件 >300 行的，提示词不得少于 100 行；>800 行的不得少于 200 行。一句话摘要、30 行以内的"结构摘要"均视为不合格输入，禁止用于生成。
- 批量模式下（AgentSwarm/多 Agent 并行），每个 Agent 的输入必须是"详细提示词文件 + 剥壳源文件"双份，禁止用一句话模板派发任务。

### 3.5 提示词质量门（生成前必过，不达标退回重提取）

进入生成前，逐项核验提示词文件；**任何一项不达标 → 退回 §3 重新提取，禁止带病进入生成**：

| 检查项 | 达标标准 |
|---|---|
| 数量守恒清单 | 存在，且每项是具体数字（"若干""多个""N 列"均不合格） |
| 下拉/单选字段 | 每个字段列出**全部选项值**（"下拉筛选""选项略"不合格） |
| 表格列 | 每张表逐列列出列名（"多列""同旧版"不合格） |
| 弹窗/抽屉 | 每个列出名称 + 内含字段 + 触发按钮 |
| 操作按钮 | 逐个列出文案 + 触发行为/跳转目标编码 |
| 业务说明段 | 存在，页面性质为五选一之一 |
| 剥壳源文件 | `prompts/<编码>-<页面名>.source.html` 存在 |
| 长度下限 | 满足 §3.4 的比例要求 |

有 JS 运行时可用时，从站点根目录运行 `node <skill>/assets/check-prompts.js prompts/<编码>-<页面名>.md [...]`，只校验本页或本批并修完所有 ✗；审计目录内全部 HTML 输入时可用 `--html-only`。纯对话工具（无运行时）则人工逐项过上表并在 progress.md 记录“质量门已过”。

> 为什么这么严：实测证明第一版质量差之后的调优会脱离 skill 管线，总消耗反而更大、质量更难控。质量门宁可多花一轮提取，也不放行残缺提示词。

## §4 内嵌组件识别（不生成独立页面）

命中以下任一信号 → 该文件是被嵌入其他页面的组件，**不得生成独立页面、不得注册菜单**：

1. 含 iframe 检测脚本：`window.self !== window.top`（最可靠信号）
2. 文件名含"内嵌-"前缀
3. 结构明显不完整（无外壳且不是完整业务页，只有一个弹窗/一个 Tab 的内容）

处理方式：先判断组件的真实形态，再整合进宿主页面：

| 组件形态 | 整合方式 |
|---|---|
| 弹窗（modal-overlay 结构） | 作为宿主页的 `data-modal` 容器嵌入，宿主页操作栏**必须同时补触发按钮**（data-action + data-modal-target） |
| 列表的某个状态 Tab | 并入宿主列表页的状态页签，并在 tableData 中**补对应状态的 mock 数据**（否则切换后表格空白） |
| 详情页的某个 Tab 内容 | 并入宿主详情页对应 Tab，确认 CSS 有对应显隐规则（如 `.sub-panel`/`.sub-panel.active`） |

不确定宿主是谁时，询问用户；无法确认时按独立页面处理并标记 `【页型待确认】`。

## §5 编码分配与兼容

- 新分配编码：按 `^[A-Z]{2}[A-Z]{2}[0-9]{2}$`（2+2+2）规则，先读 menu.config.js 顺延序号。
- **旧编码兼容**：旧系统/旧原型已有 7~8 位编码（如 `FXGLQD01`、`FXDJSH02`）时，**沿用旧编码，不强行改成 6 位**——改编码会切断与旧系统的对应关系。兼容格式为 `^[A-Z]{2,6}[0-9]{2}$`。同一站点内新旧编码可并存，但新增页面一律用 6 位规范编码。

## §6 风格确认（A/B/C 三选一）

提供旧 HTML 作为输入时，意图是"基于旧页面做新原型"，不触发门一拦截，改为风格确认：

- **A** 按高端ERP默认风格重新生成（最快，样子和旧版不像）
- **B** 提取旧页面风格写入 `theme.css`（接近旧版观感，耗时更长；theme.css 只放皮肤类规则）
- **C** 最小改动：保留旧文件结构，只注入 data-page/data-action/data-nav IR 标记（最像，但外壳重复、转 Vue 质量最低）

用户选定后立刻执行，不追问。

## §7 生成后量化自检（HTML 输入模式必做）

每页生成后，对照提示词文件的"数量守恒清单"逐项核对：

| 核对项 | 通过标准 | 不通过处理 |
|---|---|---|
| 表格列数与列名 | 与源页面完全一致 | **强制重做** |
| 查询字段数与选项值 | 完全一致 | **强制重做** |
| 状态页签数 + 每页签切换后有数据 | 完全一致且非空 | **强制重做** |
| 弹窗/抽屉数 + 每个弹窗有触发按钮 | 完全一致且可打开 | **强制重做** |
| 数据行数 | ≥ 源页面 | 补数据 |
| 图表/交互点 | 逐条落实 | 补实现，或记入 progress.md 偏差清单 |
| 业务语义 | 页面性质与"业务说明"段一致 | **强制重做** |

**缩小比警戒线**：剥离外壳后，若新页面业务内容行数 < 源页面业务内容行数的 50%，视为疑似缩水，必须重新对照数量守恒清单排查，不得直接交付。

**交付对比报告（每页必附）**：使用者手里有原页面，一定会拿产出对比——skill 必须先自证。每页交付时输出下表（数据直接来自数量守恒清单的"源页面实测 vs 生成页实测"），写进交付说明并追加到 `progress.md`：

```
| 对比项 | 源页面 | 生成页 | 结果 |
|---|---|---|---|
| 查询字段数 | 8 | 8 | ✓ |
| 表格列数 | 11 | 11 | ✓ |
| 数据行数 | 20 | 20 | ✓ |
| 状态页签数 | 5 | 5 | ✓ |
| 弹窗/抽屉数 | 3 | 3 | ✓ |
| 操作按钮数 | 4+4 | 4+4 | ✓ |
| 图表数 | 1 | 1 | ✓ |
| 交互点落实 | 6 | 6 | ✓ |
```

任何 ✗ 都必须在交付前处理（按上表的重做规则），不允许带 ✗ 交付后让使用者自己发现。

## §8 双输入生成（质量优先，三工具通用）

> 提示词再详细也是有损中间层。源页面就在磁盘上，生成时不看它就是浪费最好的参考资源。

**规则：HTML 输入模式下，生成每一页时必须同时读入两份输入——**

1. `prompts/<编码>-<页面名>.md`（提示词）：当**结构索引 + 核对清单**用，决定页面骨架和区块顺序；
2. `prompts/<编码>-<页面名>.source.html`（剥壳源文件）：当**细节兜底**用——下拉选项值、表格列宽、mock 数据原值、交互逻辑、视觉效果，直接从源文件里抄，不凭"常见做法"补位。

**大页面分块对照（按剥壳后源文件行数）**：

| 源文件行数 | 处理方式 |
|---|---|
| ≤800 行 | 整页读入，一次生成（默认路径） |
| >800 行 | **分块对照**：按区块切分源文件（查询区/统计卡/标签区/表格区/弹窗区/脚本区），逐块"读源块 → 生成对应部分"，最后拼装整页并跑一遍 §7 全量核对 |

- 800 行是按 Kimi/Codex/Claude 三工具中上下文最小者取的保守值；确认当前工具上下文充足（如 Claude）时可放宽到 1500 行，但放宽后若出现任何 §7 核对 ✗，立即回退 800 行阈值重做。
- 分块时脚本区（`<script>`）必须单独作为一块完整读入——交互逻辑最容易在切分中丢失。

**批次收紧（HTML 输入模式覆盖通用的 3~5 页/批）**：

- 每批 **2~3 页**；剥壳后 >800 行的复杂页**单页一批**；首批只做 **1 页**，交付对比报告全 ✓ 后再扩大批次。
- **质量优先于速度与 token 消耗**：禁止为省 token 跳过双输入、跳过质量门、跳过交付对比报告。第一版质量差导致的脱管线手工调优，总成本远高于生成时多读一遍源文件。
