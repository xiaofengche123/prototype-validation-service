# 多工具部署与测试准备手册（Kimi CLI → Codex → Trae）

> 用途：把 `prototype-guide` 从 Claude Code 扩到其他 3 个工具上验证。
> 本手册写给三类读者：**①你本人（操作者）②被测的 AI 工具（要让它明白自己该干什么）③团队（消除理解误差）**。
> 关联：流程见 `../SKILL.md` 与 `../references/`；测试案例见 `02-统一测试案例(危废承运商).md`。

---

## 0. 先讲原理（所有人必读，消除信息不对称）

- `prototype-guide` 的本质 = **一套 markdown 指令**（`SKILL.md` + `references/flow.md` + `references/input-intake.md` + `references/templates/*`）。
- Claude Code 把它当原生 "skill" 自动加载；**其他工具没有 "skill" 这个概念**，但它们都支持「项目级指令 / 规则 / 自定义提示」。
- 所以"在别的工具上跑 prototype-guide" = **让那个工具加载同一套 markdown，并按里面的流程工作**。
- **单一真源**：这套 markdown 不改；各工具只是用自己的机制把它"挂载"进去。

> 一句话给团队：我们不是给每个工具各写一套，而是**同一套规则，挂到不同工具上**。

---

## 1. 通用准备（你本人，测每个工具前都要做）

1. 把整个 `prototype-guide/` 目录放进**被测项目的工作目录**（让工具能读到这些文件）。
2. 准备好 `02-统一测试案例` 的「客户沟通纪要」文本。
3. 准备一个**空的输出目录**（生成原型站点用）。
4. 录屏就绪。
5. 准备好下面第 2 节的「工作交底文本」（这是让工具进入角色的钥匙）。

---

## 2. 工作交底文本（钥匙 · 工具无关 · 可复制）

> 这段文本是让**任何**工具正确扮演 prototype-guide 的"开场指令"。
> 用法见第 3 节：可作为 Codex 的 `AGENTS.md`、Trae 的 Rules、Kimi 的项目指令，或会话开头直接粘贴。

```text
你现在的角色是「产品经理原型引导助手」。请严格按本项目里 prototype-guide 的规则工作：

1. 先阅读这些文件并以它们为准：
   - prototype-guide/SKILL.md
   - prototype-guide/references/flow.md
   - prototype-guide/references/input-intake.md
   - prototype-guide/references/templates/index.md（再按选中的页型读对应模板文件）
   - 生成原型时遵循 prototype-site/SKILL.md 与 prototype-site/references/interaction-ir.md 的规则。

2. 使用者是【非技术产品经理】。全程用业务大白话，绝不向他暴露任何技术标记
   （data-action / data-nav / data-modal / 英文编码等），也不要让他看原始提示词。

3. 按 flow.md 的剧本走：开场先说明"逐页推进"工作方式并征得同意 → 收集需求（可含文档/图片）
   → 充分性检查（回吐"读到/没读到"两清单）→ 自动建议模板并用业务名确认
   → 缺口分类：技术性缺口（编码/分组/跳转占位/数据条数）自己静默补位，只有业务性缺口才"提醒+二选一"
   → 给业务语言"需求确认摘要"让他确认 → 生成【单个】原型页 + 幂等更新 menu.config.js
   → 给补位清单 + 问是否做下一页。

4. 红线：默认逐页推进，PM 坚持批量时提醒风险后照做（见 prototype-guide 治理元规则）；信息不足时永不拒绝、也不默默硬生成（走"提醒+二选一"）；
   生成的页面要真能点（查询/状态页签/弹窗/跳转意图），不得有页面级横向滚动。
5. 若输入量大（>15 个页面文件或混杂 ≥3 类材料），先做两道门诊断（见 input-intake.md 第 7 节），只诊断不拦截，等 PM 表态。
6. 批量模式每批（3-5 页）结束后更新交接文档（见 handoff-template.md），问 PM 是否开新会话。

现在请用开场白和我打招呼，开始引导我做原型。
```

> 前提：被测工具要能**读到** prototype-guide 这套文件（放进它的工作目录/上下文）。若某工具无法读本地文件，把 `SKILL.md`、`flow.md`、`input-intake.md` 和将用到的模板内容一起贴进去。

## 3. 各工具怎么挂载

### 3.1 Kimi CLI（顺序第 1，⚠ 机制需你真机首次确认）

**需要你先确认的点（这是本手册唯一高不确定项，请第一步就验证）：**
- Kimi CLI 是否读取项目根的 `AGENTS.md` 或等价的「项目指令文件」？文件名/位置是什么？
- 它能否在当前工作目录内**读取本地文件**（以便加载 prototype-guide/*）？
- 是否支持「自定义命令/自定义提示」？

**确认方法**：查 Kimi CLI 官方文档的"项目配置 / 自定义指令 / AGENTS.md / 上下文文件"章节；或在一个测试目录里放一个 AGENTS.md 写"回答时自称喵助手"，启动 Kimi 看是否生效。

**挂载方案（按确认结果择一）：**
- ✅ 若支持项目指令文件：把第 2 节文本写进它支持的文件（如 `AGENTS.md`），放到含 prototype-guide 的项目根。
- ✅ 若不支持但能读本地文件：**会话开头粘贴第 2 节文本**（万能保底），它据此去读 prototype-guide/*。
- ✅ 若连本地文件都难读：把 SKILL.md + flow.md + input-intake.md + 列表查询页.md 内容连同第 2 节文本一起粘贴。

> 万能保底：无论机制如何，「会话开头粘贴第 2 节文本」这条路一定能走通，不会因为机制没摸清而卡住测试。

### 3.2 Codex（顺序第 2，机制较明确）

- **项目指令**：在被测项目根放 `AGENTS.md`，内容 = 第 2 节文本。Codex 启动会自动加载。
- **可选自定义命令**：在 `~/.codex/prompts/` 放一个 `原型.md`（内容也用第 2 节文本），之后在 Codex 里用 `/原型` 触发。
- 它在项目目录内可读本地文件，能加载 prototype-guide/*。
- 注：以你实际的 Codex 版本/文档为准；若 `AGENTS.md` 行为有出入，退回"会话开头粘贴第 2 节文本"。

### 3.3 Trae（顺序第 3，机制需确认）

**需要你确认的点：**
- 当前 Trae 版本「项目规则 / Rules for AI」的配置入口与文件位置（是设置界面，还是 `.trae/` 下的文件）？
- 是否支持「自定义 Agent」（可设定专属 prompt + 工具）？
- 用 `#` 引用工作区文件的方式。

**挂载方案：**
- ✅ 首选：把第 2 节文本设为**项目 Rules**；用 `#` 把 prototype-guide 的文件带入上下文。
- ✅ 进阶：建一个名为「原型助手」的自定义 Agent，把第 2 节文本作为它的系统提示。
- ✅ 保底：在对话开头粘贴第 2 节文本 + `#` 引用 prototype-guide 文件。

### 3.4 DeepSeek（新增，待首次验证）

**第一步必须是能力确认**（不确认就无法定挂载方案）：

1. **文件读取能力**：DeepSeek 的当前产品形态是否能读取本地项目文件（如 `skills/prototype-guide/SKILL.md`）？
2. **Shell/工具调用能力**：是否能执行 `ls`/`find`/`dir` 等命令——这直接决定两道门的侦察方式走"自动数"还是"问用户"（见第 3.5 节降级路径）。

**确认方法**：在一个含 prototype-guide 文件的项目目录中，让 DeepSeek 尝试"列出 skills/prototype-guide/ 下所有 .md 文件"——若能列出，文件能力确认通过；让 DeepSeek 尝试"统计 test-01/ 下有多少 .html 文件"——若能跑 shell，Shell 能力确认通过。

**挂载方案（按确认结果择一）**：
- ✅ 若支持项目指令机制：把第 2 节文本写入其支持的文件。
- ✅ 保底：会话开头粘贴第 2 节文本。
- ⚠ 若连本地文件都读不了：把 SKILL.md + flow.md + input-intake.md + handoff-template.md + 模板内容连同第 2 节文本一起粘贴。

### 3.5 本次新增机制的按工具注意事项

> 以下为 05 号设计方案中新增机制（输入分诊、两道门、拆分、批次交接）在各工具上的适配要点，写进此手册统一管理。

**Claude Code（基线）**：
- 两道门的侦察直接用 Bash/Glob 等工具跑出精确数字，无需降级。
- 交接文档天然跨会话生效（文件 I/O 可靠），是完整验证基线。

**Kimi**：
- 交接文档必须是独立文件而非依赖对话记忆——Kimi 实测中出现过异常退出，文件是唯一可靠的状态载体。
- 工作交底文本里两道门的规则要写成简短祈使句中文，不要长段抽象描述（参考 `原型skill使用与改进建议.txt`："Kimi 对中文长规则黏着度更好"）。
- Shell 能力待确认：若无法跑 `ls`/`find`，两道门侦察降级为直接问用户要数字。

**Codex**：
- 两道门的侦察可直接用 shell 命令，机制明确，风险最低。
- `AGENTS.md` 里已包含新增机制的编排段落（见工具无关编排提示词），无需额外操作。

**DeepSeek**：
- 先完成第 3.4 节的能力确认，再决定走哪条降级路径（见下）。

### 3.6 降级路径：工具没有可靠的文件/Shell 能力时

若目标工具经确认不具备 Shell/文件列举能力（典型如 DeepSeek 在某些托管环境下，或 Kimi 某些版本），两道门的资源过载检查退化：

> "为了不让处理过程中途卡死，能不能先告诉我：这次一共涉及几个页面文件？大概多少字/多少张截图？"

不要凭模型自己数上下文里贴的内容——因为凭感觉容易数错。这条降级路径是每个新工具接入前必须先确认的第一件事：**先探测工具的文件/Shell 能力，再决定两道门的侦察方式走"自动数"还是"问用户"。**

## 4. 每个工具的测试（统一用 02 案例）

1. 按本工具的挂载方案让它进入角色。
2. 走 `02-统一测试案例` 的输入与标准答案。
3. 用 02 的「验收点」打分，并填 02 末尾的「横向对比记录表」一行。
4. **Kimi 这一轮按要求录屏**（和 Claude 同案例，便于团队对比）。

---

## 5. 出问题怎么记录（反馈给开发）

每个问题记：**哪个工具 / 第几步 / 你说了什么 / 它实际怎么做 / 期望怎么做 / 录屏时间点**。
特别关注三类高发差异：
- **没遵守"不暴露技术词"**（弱模型容易把 data-action 等吐出来）。
- **选错模板**（AI 选型不准）。
- **该静默补的技术缺口反而来问你**（缺口分类没执行好）。

这些大多能通过强化第 2 节文本或对应模板措辞来修，少数是工具能力限制。

## 6. 建议的推进节奏

1. 你先按 `01-测试方案` 录完 **Claude 版**（基线，证明流程本身对）。
2. 进 **Kimi**：先花 10 分钟做 3.1 的「机制确认」，确认不了就直接用"粘贴第 2 节文本"保底开测，按 02 案例跑 + 录屏。
3. 再 **Codex**、**Trae**，各填一行对比记录。
4. 把三个工具的对比记录 + 问题清单给我，我据此修 skill / 调整第 2 节文本。
