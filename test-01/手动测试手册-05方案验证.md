## 三、用例组 G1 · 门一：目标兼容性检查

### G1-1 要求像素级复刻

- **操作**：口述（**不留文件**，避免附带文件干扰门二判断），说：
  > 我有个旧原型页面，帮我在新站点里做得一模一样，像素级的。
- **期望（按顺序）**：
  1. [ ] 识别到"像素级还原"信号，**在生成前**停下来
  2. [ ] 给出 A/B/C 三条路，每条路的代价讲清楚（A 默认风格不像 / B 提炼风格接近但更耗时 / C 直接复用丢 IR）
  3. [ ] 格式符合诊断报告四段式（我看到 / 会带来的问题 / 我的建议 / 需要你确认）
  4. [ ] 你说"选 B"之后立刻执行，不追问"你确定吗"
  5. [ ] 全程**不出现** CSS/技术术语（用大白话说"样子""风格"，不说"theme.css""CSS 变量"）
  6. [ ] **不触发门二诊断报告**（没有文件输入，不应做资源过载检查）
- **产物检查**：若选 B，站点 `assets/theme.css` 应存在且无布局规则。

### G1-2 要求换 UI 风格/组件库（门一第二触发信号）

- **操作**：口述，不给文件：
  > 帮我把这套原型换成 Ant Design 的风格，不要现在这个绿星链通的样式。
- **期望**：
  1. [ ] 识别到"换 UI 风格"信号，**在生成前**停下来
  2. [ ] 说明"临时换风格只能做到接近，正式换需要另立项集成组件库"
  3. [ ] 给出可执行选项，不讲"做不到"就拒绝
  4. [ ] 你说"那就尽量接近吧"后立刻执行

## 四、用例组 G2 · 门二：资源过载检查

### G2-1 批量旧原型目录触发高风险

- **操作**：把整个旧原型目录（34 个 html）+ PRD + 编码表一起给，说：
  > 这是我们的需求和旧原型，帮我重新做。
- **文件传递方式**：拖拽整个 `审计监督模型` 文件夹到对话中（或逐一附上 3 类材料：PRD 文件 + 编码表文件 + 旧原型目录路径）。不同附加方式可能影响 AI 的侦察行为，**记录你用的方式**便于后续复现。
- **期望（按顺序）**：
  1. [ ] **先侦察再说话**——用 `ls`/`find` 数出文件数量，不先把 34 个文件全读进上下文
  2. [ ] 输出诊断报告（四段式），**必须包含具体数字**（"34 个页面文件"），不能说"感觉很多"
  3. [ ] 主动给拆分建议——"要不要我先把这些拆成独立的小提示词？"
  4. [ ] **不自行决定拒绝**——选项里有"全部做"的路
- **违规判定**：
  - 先读了大量文件全文才报告 = 违规（没执行"先侦察不读全量"）
  - 说"太多了做不了"直接拒绝 = 违规（违反治理元规则）

### G2-2 用户说"别管了，全部做"

- **操作**：接 G2-1，在诊断报告后说：
  > 别管了，全部做，一页页做就行。
- **期望**：
  1. [ ] 不再劝，直接照做
  2. [ ] 交付说明里有"按您的要求，已提示风险后执行"留痕

### G2-3 单页输入不触发

- **操作**：只给 PRD，说"帮我做任务管理列表这一页"。
- **期望**：
  - [ ] **不触发**两道门诊断报告（1 类材料、目标明确、无矛盾）

### G2-4 边界值：恰好 15 个页面文件（中/高风险分界线）

- **操作**：从旧原型目录中精确挑选 15 个 html 文件（不附带 PRD 和编码表），说：
  > 帮我把这些旧原型页面重新做成新站点。
- **关键观察**：`input-intake.md` 第 7.2 节阈值为 ≤5 低风险 / 6~15 中风险 / >15 高风险。恰好 15 属于"中风险"上限，**不应**触发高风险诊断报告。
- **期望**：
  - [ ] 执行侦察（数文件数量），但不输出"高风险"诊断报告
  - [ ] 可正常进入逐页流程，不强制要求拆分
- **违规判定**：对恰好 15 个文件输出"资源过载高风险"诊断 = 阈值判断偏了一档。

### G2-5 边界值：恰好 2 类材料 + 6 个文件（中风险上限）

- **操作**：给 PRD（1 类）+ 6 个旧原型 html（第 2 类），说：
  > 帮我做这几个页面。
- **期望**：
  - [ ] 识别到 2 类材料，但不触发高风险诊断（需 ≥3 类才触发）
  - [ ] 正常进入逐页流程

## 五、用例组 SPLIT · 拆分与批次

### SPLIT-1 完整拆分→批次执行

- **操作**：同 G2-1 输入，在诊断报告后说：
  > 行，你先帮我拆成小提示词文件吧。
- **期望（按顺序）**：
  1. [ ] 若站点目录尚不存在，先建站点骨架（复制 `assets/framework/`），再在站点目录下创建 `prompts/` 子目录
  2. [ ] 产出 N 份独立提示词文件到 `<站点目录>/prompts/` 下（如 `RWGL01-任务管理列表.md`）
  3. [ ] 每份约 30 行，包含：页面编码、名称、类型、菜单归属、查询字段、表格列、操作、交互要求
  4. [ ] 列出清单让你过目确认，等你勾选本批要做的页
  5. [ ] 你说"先做前 3 页"后，逐页执行——每页生成前重新读取该页的提示词文件（而非凭记忆）

### SPLIT-2 用户跳过拆分

- **操作**：同 G2-1，在诊断报告后说：
  > 别拆了，你看着办吧，全部一起做。
- **期望**：
  1. [ ] 提醒风险（一句话，不重复）
  2. [ ] 你确认后照做
  3. [ ] 交付说明留痕

## 六、用例组 HANDOFF · 交接文档与会话接续

### HANDOFF-1 批次结束产生交接文档

- **操作**：执行 SPLIT-1 的前 3 页后（或主动提出"这批先到这"）。
- **期望**：
  1. [ ] 主动产出或更新交接文档（路径应在站点目录或约定位置）
  2. [ ] 交接文档包含：已完成清单、踩坑记录、待办清单、下一步建议、文件改动概览
  3. [ ] 主动问"要不要开个新会话继续？"

### HANDOFF-2 新会话读交接文档接续

- **操作**：**新开一个会话**，只把 HANDOFF-1 产出的交接文档丢进去，说：
  > 这是上一批的进度，你接着做剩下的。
- **期望**：
  1. [ ] 根据交接文档的待办清单和已完成页面，直接开始做下一页
  2. [ ] **不重新问一遍需求**（不需要重新确认 PRD 的业务口径）
  3. [ ] 知道之前的站点路径，不做重复的站点初始化
- **判据细化（区分"合理追问"和"重新问需求"）**：
  - ✅ 合理（不判失败）：交接文档中「站点路径」「待办清单」等关键字段为空 → AI 追问"站点路径是什么？"或"还有哪些页面没做？"——这是上一轮没填好，不是本轮 AI 的问题。**记录为交接文档质量问题，不记为本用例失败。**
  - ❌ 违规（判失败）：交接文档信息完整，AI 仍然问"你想做什么页面？""需求文档在哪？"——说明没有真正读取交接文档的内容。

## 七、用例组 META · 治理元规则验证

### META-1 偏好不啰嗦

- **操作**：在任一用例中说：
  > 示例数据给我放 20 条；新增按钮的字改成"登记"。
- **期望**：
  - [ ] 直接照做，**不触发**风险说教

### META-2 风险只讲一次

- **操作**：在 G2-1 诊断报告后说"全部做"→ 如果 AI 再次提醒风险，记违规。在 G1-1 选 B 后 → 如果 AI 再次确认"你真的要选 B 吗"，记违规。
- **违规判定**：任何已经明确表态后的再次劝说 = 违规。

### META-3 脏活自己扛

- **操作**：在 SPLIT-1 中观察——skill 是否主动提出拆分、是否把拆分文件格式都给你准备好、是否主动更新交接文档。
- **期望**：
  - [ ] 拆分、交接文档这些工序由 AI 主动完成，不需要你说"你帮我建个文件记录一下进度"

## 九、通用产物检查

每个生成了页面的用例都做：

**文件级**：
- [ ] `data-page="<英文编码>"` 在页面根节点
- [ ] 业务页无 `<style>` 标签、无外壳标记（`top-header`/`sidebar`/`page-tabs`）
- [ ] `menu.config.js` code 无重复
- [ ] 表格数据驱动（JS 数组 + render）

**浏览器级**：
- [ ] `document.scrollingElement.scrollWidth <= document.scrollingElement.clientWidth` 为 true
- [ ] 窄表无多余横滚；同排卡片底边齐平；无大片空白
- [ ] 底部操作栏固定不动
- [ ] 点按钮有响应

## 十、测试顺序建议

```
S1 → S2 (老流程回归)
   ↓
G1-1 → G1-2 → G2-3 → G2-4 → G2-5 → G2-1 → G2-2 (门一全部 + 门二先边界再高风险)
   ↓
SPLIT-1 → SPLIT-2 (拆分)
   ↓
HANDOFF-1 → HANDOFF-2 (交接 + 接续，HANDOFF-2 需新开会话)
   ↓
META-1 → META-2 → META-3 (元规则边界)
   ↓
STYLE-1 → STYLE-2 (组件库回归)
```

顺序理由：S1/S2 先确认没改坏；G 组先跑边界值（G2-3/4/5 轻量）再跑高风险场景（G2-1 重）；G2-2 复用 G2-1 会话；HANDOFF-2 需单独新开会话，安排在中段方便后续用例继续复用；STYLE 做完前面自然有产物可用。
