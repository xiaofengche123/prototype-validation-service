# prototype-guide / prototype-site 改造后手动测试手册（Kimi）

> **测试目标**：验证本次新增的「现成原型页输入 → A/B 双路线」机制和「偏离路线处理」机制，同时确认原有功能未回归。
> **测试工具**：Kimi CLI（skill 加载方式见 `skills/prototype-guide/测试与交付/03-多工具准备手册.md`）。
> **结果归档**：全部放本目录（`test-01/result/`），约定见文末「结果记录规范」。

---

## 一、测试前准备

1. **确认 Kimi 能读到两个 skill**：`skills/prototype-guide/` 和 `skills/prototype-site/` 在 Kimi 的工作目录/上下文内。
2. **测试素材**（现成的，不用另造）：
   - PRD：`test-01/审计监督模型_2026年6月30日/审计监督模型/绿星智审-审计监督模型_PRD_需求规格说明书_V1.md`
   - 编码表：`test-01/审计监督模型_2026年6月30日/审计监督模型/应用模块划分与编码.md`
   - 旧版原型页：同目录下各子目录的 `.html` 文件（任务中心/模型中心/风险管理/首页）
3. **每个用例用全新的目标站点目录**，按用例编号建：`test-01/result/<用例编号>/site/`。用例之间不共用站点，避免互相污染。
4. 测试时**扮演不懂技术的产品经理**：只说业务话，不主动提"IR/data-action/theme.css"这些词——这些词如果从 AI 嘴里对你说出来，本身就算一个缺陷。

## 二、用例组 R · 回归测试（原有功能不受影响）

### R1 口述做单页（老流程原样）

- **操作**：不给任何文件，直接口述，例如：
  > 我要做一个"供应商证照管理"页面，能按供应商名称和证照状态查询，列表显示供应商、证照名称、有效期、状态，能新增证照。
  （也可直接套用 `测试与交付/05-手动测试案例-供应商证照(Kimi).md`）
- **期望**：
  - [ ] 开场解释"一次只做一页"并征得同意
  - [ ] 回吐"✅读到了 / ❓没读到"两清单
  - [ ] 用业务化名称建议模板（如"数据清单页"），不出现内部叫法
  - [ ] **全程没有**被问到 A/B 路线（没有 html 输入，不该触发新流程）
  - [ ] 生成后给补位清单，问是否做下一页
- **产物检查**：见「四、通用产物检查」的默认风格组。

### R2 纯文档输入（PRD，不带 html）

- **操作**：只丢 PRD 和编码表两个 `.md`，说：
  > 这是我们的需求文档和模块编码，帮我先做"任务管理列表"这一页。
- **期望**：
  - [ ] 正常走解析 → 两清单 → 模板确认 → 生成，**不触发** A/B 路线询问
  - [ ] 页面编码参考了编码表（如 RWGL 前缀）

## 三、用例组 A/B · 双路线测试（本次改造核心）

### A1 原型页输入 → 选路线 A（平台默认风格）

- **操作**：把 PRD + 编码表 + 旧原型页目录一起给 Kimi，说：
  > 这是我们的需求文档和以前做的一套原型页面，帮我重新做原型。
- **期望（按顺序发生）**：
  1. [ ] **盘点**：数出页面数量、逐页说出用途，让你确认"这些页都认到了"
  2. [ ] **风格差异分析**：用大白话说出旧原型和默认风格的差异（顶栏/卡片/表头之类），**不出现 CSS 术语**
  3. [ ] **主动抛出 A/B 二选一**，两条路的好处和代价都讲了，并给了推荐——**不能不问就自己定**
  4. [ ] 你回答"选 A"后，走正常逐页流程（模板确认 → 生成第一页 → 补位清单 → 问下一页）
  5. [ ] 混合输入处理正确：业务口径以 PRD 为准，页面长相参照对应旧原型页
- **产物检查**（默认风格组，见第四节），另加：
  - [ ] 站点 `assets/` 下**没有** `theme.css`
  - [ ] 旧原型的 `.html` 文件**没有**被复制进站点

### B1 原型页输入 → 选路线 B（按旧原型提炼统一风格）

- **操作**：同 A1 输入，走到 A/B 询问时回答：
  > 选 B，我想保留原来那套的样子。
- **期望（按顺序发生）**：
  1. [ ] 让你**指定 1~2 页作为视觉标准**（基准页），例如你指定"1.任务驾驶舱.html"
  2. [ ] 生成站点级 `assets/theme.css`，并**先做 1 个样板页给你看**，等你认可
  3. [ ] 你说"可以"之后才继续做后面的页；你如果说"顶栏颜色不对"，它应调整 theme.css 而不是往页面里写样式
- **产物检查**（在默认风格组基础上）：
  - [ ] 站点 `assets/theme.css` 存在，文件头部有注释记录"哪个值来自哪个基准页"
  - [ ] theme.css 内容只有颜色/字体/圆角/阴影/皮肤类覆盖，**没有** display/grid/width/position 等布局规则，**没有**只服务某一页的选择器
  - [ ] `index.html` 和每个业务页里 `theme.css` 的 `<link>` 都紧跟在 `app.css` 之后
  - [ ] 业务页依然**没有**内联 `<style>`（风格全部走 theme.css）
  - [ ] **skill 目录没被改**：`skills/prototype-site/assets/framework/` 下不应出现 theme.css 或任何修改（可用 `git status` 看 skills 目录是否干净）
  - [ ] 页面视觉肉眼对比接近基准页（主色、顶栏、表头底色）

## 四、用例组 D · 偏离路线处理测试

> 三步走的判定标准：**提醒（风险一次讲完整）→ 你确认后照做（不再反复劝）→ 交付时补位清单里留痕**。拒绝执行 = 缺陷；不提醒直接照做 = 缺陷；确认后还在反复劝 = 缺陷。

### D1 结构性偏离：要求照抄旧代码

- **操作**：在 A1/B1 流程中途说：
  > 别重新做了，直接把我原来那些 html 复制过来改改就行。
- **期望**：
  - [ ] 讲清风险（旧页面自带外壳和独立样式，进站点会冲突、以后没法统一改）
  - [ ] 你坚持后照做，交付说明里有「按您的要求偏离了推荐做法」记录
  - [ ] 你改口"那还是按你推荐的来"后能顺利回到正轨

### D2 结构性偏离：要求每页独立、自带样式

- **操作**：
  > 我不要什么共享样式，每个页面独立生成，样式写在页面里。
- **期望**：同 D1 三步走；风险提醒里应点到"改一处公共样式要改 N 遍"和"转正式开发每页单独对视觉"这两点（大意到位即可）。

### D3 结构性偏离：要求跳过逐页确认、全部一次做完

- **操作**：
  > 别一页页问我了，你把 PRD 里的页面全部一次做完。
- **期望**：
  - [ ] 提醒风险（跳过确认，返工风险归你），你确认后照做
  - [ ] **注意观察并记录**：批量做完后各页质量如何、后面的页是否明显劣化——这组数据是我们下一步"批量模式"改造的依据，请重点截图留存。

### D4 小偏好：不应被啰嗦

- **操作**：在任一用例中说：
  > 示例数据给我放 20 条；另外新增按钮的字改成"登记"。
- **期望**：
  - [ ] 直接照做，**没有**风险说教（小偏好不走三步）

## 五、通用产物检查（每个生成了页面的用例都做）

**文件级**（在站点目录下检查）：

- [ ] 页面路径是 `pages/<分组中文名>/<页面中文名>.html`
- [ ] 页面根节点有 `data-page="<英文编码>"`；按钮有 `data-action`，跳转用 `data-nav`
- [ ] 业务页无内联 `<style>`、无顶栏/侧栏/页签等外壳标记（可全局搜索 `<style` 和 `top-header` 验证）
- [ ] `menu.config.js` 每个 code 只有一行，file 指向真实文件
- [ ] 表格是数据驱动（JS 数组 + render），没有手写死的 `<tr>` 数据行

**浏览器级**（用浏览器打开站点 `index.html`，从菜单进入每个新页）：

- [ ] 页面在外壳里正常渲染，无页面级横向滚动条（F12 控制台执行
  `document.scrollingElement.scrollWidth <= document.scrollingElement.clientWidth` 应为 true）
- [ ] 窄表格没有多余横向滚动条；同排卡片底边基本齐平；无大片空白
- [ ] 有保存/提交类操作的页面，底部操作栏固定不动
- [ ] 点按钮有响应（弹窗/提示/跳转，或"待实现"提示，不能毫无反应）
